222 r"""specify a fixed VALUES clause for an INSERT statement, or the SET 

223 clause for an UPDATE. 

224 

225 Note that the :class:`.Insert` and :class:`.Update` constructs support 

226 per-execution time formatting of the VALUES and/or SET clauses, 

227 based on the arguments passed to :meth:`.Connection.execute`. 

228 However, the :meth:`.ValuesBase.values` method can be used to "fix" a 

229 particular set of parameters into the statement. 

230 

231 Multiple calls to :meth:`.ValuesBase.values` will produce a new 

232 construct, each one with the parameter list modified to include 

233 the new parameters sent. In the typical case of a single 

234 dictionary of parameters, the newly passed keys will replace 

235 the same keys in the previous construct. In the case of a list-based 

236 "multiple values" construct, each new list of values is extended 

237 onto the existing list of values. 

238 

239 :param \**kwargs: key value pairs representing the string key 

240 of a :class:`.Column` mapped to the value to be rendered into the 

241 VALUES or SET clause:: 

242 

243 users.insert().values(name="some name") 

244 

245 users.update().where(users.c.id==5).values(name="some name") 

246 

247 :param \*args: As an alternative to passing key/value parameters, 

248 a dictionary, tuple, or list of dictionaries or tuples can be passed 

249 as a single positional argument in order to form the VALUES or 

250 SET clause of the statement. The forms that are accepted vary 

251 based on whether this is an :class:`.Insert` or an :class:`.Update` 

252 construct. 

253 

254 For either an :class:`.Insert` or :class:`.Update` construct, a 

255 single dictionary can be passed, which works the same as that of 

256 the kwargs form:: 

257 

258 users.insert().values({"name": "some name"}) 

259 

260 users.update().values({"name": "some new name"}) 

261 

262 Also for either form but more typically for the :class:`.Insert` 

263 construct, a tuple that contains an entry for every column in the 

264 table is also accepted:: 

265 

266 users.insert().values((5, "some name")) 

267 

268 The :class:`.Insert` construct also supports being passed a list 

269 of dictionaries or full-table-tuples, which on the server will 

270 render the less common SQL syntax of "multiple values" - this 

271 syntax is supported on backends such as SQLite, PostgreSQL, MySQL, 

272 but not necessarily others:: 

273 

274 users.insert().values([ 

275 {"name": "some name"}, 

276 {"name": "some other name"}, 

277 {"name": "yet another name"}, 

278 ]) 

279 

280 The above form would render a multiple VALUES statement similar to:: 

281 

282 INSERT INTO users (name) VALUES 

283 (:name_1), 

284 (:name_2), 

285 (:name_3) 

286 

287 It is essential to note that **passing multiple values is 

288 NOT the same as using traditional executemany() form**. The above 

289 syntax is a **special** syntax not typically used. To emit an 

290 INSERT statement against multiple rows, the normal method is 

291 to pass a multiple values list to the :meth:`.Connection.execute` 

292 method, which is supported by all database backends and is generally 

293 more efficient for a very large number of parameters. 

294 

295 .. seealso:: 

296 

297 :ref:`execute_multiple` - an introduction to 

298 the traditional Core method of multiple parameter set 

299 invocation for INSERTs and other statements. 

300 

301 .. versionchanged:: 1.0.0 an INSERT that uses a multiple-VALUES 

302 clause, even a list of length one, 

303 implies that the :paramref:`.Insert.inline` flag is set to 

304 True, indicating that the statement will not attempt to fetch 

305 the "last inserted primary key" or other defaults. The 

306 statement deals with an arbitrary number of rows, so the 

307 :attr:`.ResultProxy.inserted_primary_key` accessor does not 

308 apply. 

309 

310 .. versionchanged:: 1.0.0 A multiple-VALUES INSERT now supports 

311 columns with Python side default values and callables in the 

312 same way as that of an "executemany" style of invocation; the 

313 callable is invoked for each row. See :ref:`bug_3288` 

314 for other details. 

315 

316 The :class:`.Update` construct supports a special form which is a 

317 list of 2-tuples, which when provided must be passed in conjunction 

318 with the 

319 :paramref:`~sqlalchemy.sql.expression.update.preserve_parameter_order` 

320 parameter. 

321 This form causes the UPDATE statement to render the SET clauses 

322 using the order of parameters given to :meth:`.Update.values`, rather 

323 than the ordering of columns given in the :class:`.Table`. 

324 

325 .. versionadded:: 1.0.10 - added support for parameter-ordered 

326 UPDATE statements via the 

327 :paramref:`~sqlalchemy.sql.expression.update.preserve_parameter_order` 

328 flag. 

329 

330 .. seealso:: 

331 

332 :ref:`updates_order_parameters` - full example of the 

333 :paramref:`~sqlalchemy.sql.expression.update.preserve_parameter_order`

335 

336 .. seealso:: 

337 

338 :ref:`inserts_and_updates` - SQL Expression 

339 Language Tutorial 

340 

341 :func:`~.expression.insert` - produce an ``INSERT`` statement

401 def return_defaults(self, *cols): 

402 """Make use of a :term:`RETURNING` clause for the purpose 

403 of fetching server-side expressions and defaults. 

404 

405 E.g.:: 

406 

407 stmt = table.insert().values(data='newdata').return_defaults() 

408 

409 result = connection.execute(stmt) 

410 

411 server_created_at = result.returned_defaults['created_at'] 

412 

413 When used against a backend that supports RETURNING, all column 

414 values generated by SQL expression or server-side-default will be 

415 added to any existing RETURNING clause, provided that 

416 :meth:`.UpdateBase.returning` is not used simultaneously. The column 

417 values will then be available on the result using the 

418 :attr:`.ResultProxy.returned_defaults` accessor as a dictionary, 

419 referring to values keyed to the :class:`.Column` object as well as 

420 its ``.key``. 

421 

422 This method differs from :meth:`.UpdateBase.returning` in these ways: 

423 

424 1. :meth:`.ValuesBase.return_defaults` is only intended for use with 

425 an INSERT or an UPDATE statement that matches exactly one row. 

426 While the RETURNING construct in the general sense supports 

427 multiple rows for a multi-row UPDATE or DELETE statement, or for 

428 special cases of INSERT that return multiple rows (e.g. INSERT from 

429 SELECT, multi-valued VALUES clause), 

430 :meth:`.ValuesBase.return_defaults` is intended only for an 

431 "ORM-style" single-row INSERT/UPDATE statement. The row returned 

432 by the statement is also consumed implicitly when 

433 :meth:`.ValuesBase.return_defaults` is used. By contrast, 

434 :meth:`.UpdateBase.returning` leaves the RETURNING result-set 

435 intact with a collection of any number of rows. 

436 

437 2. It is compatible with the existing logic to fetch auto-generated 

438 primary key values, also known as "implicit returning". Backends 

439 that support RETURNING will automatically make use of RETURNING in 

440 order to fetch the value of newly generated primary keys; while the 

441 :meth:`.UpdateBase.returning` method circumvents this behavior, 

442 :meth:`.ValuesBase.return_defaults` leaves it intact. 

443 

444 3. It can be called against any backend. Backends that don't support 

445 RETURNING will skip the usage of the feature, rather than raising 

446 an exception. The return value of 

447 :attr:`.ResultProxy.returned_defaults` will be ``None`` 

448 

449 :meth:`.ValuesBase.return_defaults` is used by the ORM to provide 

450 an efficient implementation for the ``eager_defaults`` feature of 

451 :func:`.mapper`. 

452 

453 :param cols: optional list of column key names or :class:`.Column` 

454 objects. If omitted, all column expressions evaluated on the server 

455 are added to the returning list. 

456 

457 .. versionadded:: 0.9.0 

458 

459 .. seealso:: 

460 

461 :meth:`.UpdateBase.returning` 

462 

463 :attr:`.ResultProxy.returned_defaults` 

464 

465 """

646 

647 E.g.::

652 values(name='user #5') 

653 

654 Similar functionality is available via the 

655 :meth:`~.TableClause.update` method on 

656 :class:`.Table`:: 

657 

658 stmt = users.update().\ 

659 where(users.c.id==5).\ 

660 values(name='user #5') 

661 

662 :param table: A :class:`.Table` object representing the database 

663 table to be updated. 

664 

665 :param whereclause: Optional SQL expression describing the ``WHERE`` 

666 condition of the ``UPDATE`` statement. Modern applications 

667 may prefer to use the generative :meth:`~Update.where()` 

668 method to specify the ``WHERE`` clause. 

669 

670 The WHERE clause can refer to multiple tables. 

671 For databases which support this, an ``UPDATE FROM`` clause will 

672 be generated, or on MySQL, a multi-table update. The statement 

673 will fail on databases that don't have support for multi-table 

674 update statements. A SQL-standard method of referring to 

675 additional tables in the WHERE clause is to use a correlated 

676 subquery:: 

677 

678 users.update().values(name='ed').where( 

679 users.c.name==select([addresses.c.email_address]).\ 

680 where(addresses.c.user_id==users.c.id).\ 

681 as_scalar() 

682 ) 

683 

684 :param values: 

685 Optional dictionary which specifies the ``SET`` conditions of the 

686 ``UPDATE``. If left as ``None``, the ``SET`` 

687 conditions are determined from those parameters passed to the 

688 statement during the execution and/or compilation of the 

689 statement. When compiled standalone without any parameters, 

690 the ``SET`` clause generates for all columns. 

691 

692 Modern applications may prefer to use the generative 

693 :meth:`.Update.values` method to set the values of the 

694 UPDATE statement. 

695 

696 :param inline: 

697 if True, SQL defaults present on :class:`.Column` objects via 

698 the ``default`` keyword will be compiled 'inline' into the statement 

699 and not pre-executed. This means that their values will not 

700 be available in the dictionary returned from 

701 :meth:`.ResultProxy.last_updated_params`. 

702 

703 :param preserve_parameter_order: if True, the update statement is 

704 expected to receive parameters **only** via the 

705 :meth:`.Update.values` method, and they must be passed as a Python 

706 ``list`` of 2-tuples. The rendered UPDATE statement will emit the SET 

707 clause for each referenced column maintaining this order. 

708 

709 .. versionadded:: 1.0.10 

710 

711 .. seealso:: 

712 

713 :ref:`updates_order_parameters` - full example of the 

714 :paramref:`~.update.preserve_parameter_order` flag 

715 

716 If both ``values`` and compile-time bind parameters are present, the 

717 compile-time bind parameters override the information specified 

718 within ``values`` on a per-key basis. 

719 

720 The keys within ``values`` can be either :class:`.Column` 

721 objects or their string identifiers (specifically the "key" of the 

722 :class:`.Column`, normally but not necessarily equivalent to 

723 its "name"). Normally, the 

724 :class:`.Column` objects used here are expected to be 

725 part of the target :class:`.Table` that is the table 

726 to be updated. However when using MySQL, a multiple-table 

727 UPDATE statement can refer to columns from any of 

728 the tables referred to in the WHERE clause. 

729 

730 The values referred to in ``values`` are typically: 

731 

732 * a literal data value (i.e. string, number, etc.) 

733 * a SQL expression, such as a related :class:`.Column`, 

734 a scalar-returning :func:`.select` construct, 

735 etc. 

736 

737 When combining :func:`.select` constructs within the values 

738 clause of an :func:`.update` construct, 

739 the subquery represented by the :func:`.select` should be 

740 *correlated* to the parent table, that is, providing criterion 

741 which links the table inside the subquery to the outer table 

742 being updated:: 

743 

744 users.update().values( 

745 name=select([addresses.c.email_address]).\ 

746 where(addresses.c.user_id==users.c.id).\ 

747 as_scalar() 

748 ) 

749 

750 .. seealso:: 

751 

752 :ref:`inserts_and_updates` - SQL Expression 

753 Language Tutorial 

754
