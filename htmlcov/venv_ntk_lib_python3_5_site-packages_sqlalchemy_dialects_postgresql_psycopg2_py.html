14psycopg2 Connect Arguments 

15----------------------------------- 

16 

17psycopg2-specific keyword arguments which are accepted by 

18:func:`.create_engine()` are: 

19 

20* ``server_side_cursors``: Enable the usage of "server side cursors" for SQL 

21 statements which support this feature. What this essentially means from a 

22 psycopg2 point of view is that the cursor is created using a name, e.g. 

23 ``connection.cursor('some name')``, which has the effect that result rows 

24 are not immediately pre-fetched and buffered after statement execution, but 

25 are instead left on the server and only retrieved as needed. SQLAlchemy's 

26 :class:`~sqlalchemy.engine.ResultProxy` uses special row-buffering 

27 behavior when this feature is enabled, such that groups of 100 rows at a 

28 time are fetched over the wire to reduce conversational overhead. 

29 Note that the :paramref:`.Connection.execution_options.stream_results` 

30 execution option is a more targeted 

31 way of enabling this mode on a per-execution basis. 

32 

33* ``use_native_unicode``: Enable the usage of Psycopg2 "native unicode" mode 

34 per connection. True by default. 

35 

36 .. seealso:: 

37 

38 :ref:`psycopg2_disable_native_unicode` 

39 

40* ``isolation_level``: This option, available for all PostgreSQL dialects, 

41 includes the ``AUTOCOMMIT`` isolation level when using the psycopg2 

42 dialect. 

43 

44 .. seealso:: 

45 

46 :ref:`psycopg2_isolation_level` 

47 

48* ``client_encoding``: sets the client encoding in a libpq-agnostic way, 

49 using psycopg2's ``set_client_encoding()`` method. 

50 

51 .. seealso:: 

52 

53 :ref:`psycopg2_unicode` 

54 

55* ``executemany_mode``, ``executemany_batch_page_size``, 

56 ``executemany_values_page_size``: Allows use of psycopg2 

57 extensions for optimizing "executemany"-stye queries. See the referenced 

58 section below for details. 

59 

60 .. seealso:: 

61 

62 :ref:`psycopg2_executemany_mode` 

63 

64* ``use_batch_mode``: this is the previous setting used to affect "executemany" 

65 mode and is now deprecated.

68Unix Domain Connections 

69------------------------ 

70 

71psycopg2 supports connecting via Unix domain connections. When the ``host`` 

72portion of the URL is omitted, SQLAlchemy passes ``None`` to psycopg2, 

73which specifies Unix-domain communication rather than TCP/IP communication:: 

74 

75 create_engine("postgresql+psycopg2://user:password@/dbname") 

76 

77By default, the socket file used is to connect to a Unix-domain socket 

78in ``/tmp``, or whatever socket directory was specified when PostgreSQL 

79was built. This value can be overridden by passing a pathname to psycopg2, 

80using ``host`` as an additional keyword argument:: 

81 

82 create_engine("postgresql+psycopg2://user:password@/dbname?host=/var/lib/postgresql") 

83 

84.. seealso:: 

85 

86 `PQconnectdbParams \ 

87 <http://www.postgresql.org/docs/9.1/static/libpq-connect.html#LIBPQ-PQCONNECTDBPARAMS>`_

89Empty DSN Connections / Environment Variable Connections 

90--------------------------------------------------------- 

91 

92The psycopg2 DBAPI can connect to PostgreSQL by passing an empty DSN to the 

93libpq client library, which by default indicates to connect to a localhost 

94PostgreSQL database that is open for "trust" connections. This behavior can be 

95further tailored using a particular set of environment variables which are 

96prefixed with ``PG_...``, which are consumed by ``libpq`` to take the place of 

97any or all elements of the connection string. 

98 

99For this form, the URL can be passed without any elements other than the 

100initial scheme:: 

101 

102 engine = create_engine('postgresql+psycopg2://') 

103 

104In the above form, a blank "dsn" string is passed to the ``psycopg2.connect()`` 

105function which in turn represents an empty DSN passed to libpq. 

106 

107.. versionadded:: 1.3.2 support for parameter-less connections with psycopg2. 

108 

109.. seealso:: 

110 

111 `Environment Variables\ 

112 <https://www.postgresql.org/docs/current/libpq-envars.html>`_ - 

113 PostgreSQL documentation on how to use ``PG_...`` 

114 environment variables for connections.

118Per-Statement/Connection Execution Options 

119------------------------------------------- 

120 

121The following DBAPI-specific options are respected when used with 

122:meth:`.Connection.execution_options`, :meth:`.Executable.execution_options`, 

123:meth:`.Query.execution_options`, in addition to those not specific to DBAPIs: 

124 

125* ``isolation_level`` - Set the transaction isolation level for the lifespan 

126 of a :class:`.Connection` (can only be set on a connection, not a statement 

127 or query). See :ref:`psycopg2_isolation_level`. 

128 

129* ``stream_results`` - Enable or disable usage of psycopg2 server side 

130 cursors - this feature makes use of "named" cursors in combination with 

131 special result handling methods so that result rows are not fully buffered. 

132 If ``None`` or not set, the ``server_side_cursors`` option of the 

133 :class:`.Engine` is used. 

134 

135* ``max_row_buffer`` - when using ``stream_results``, an integer value that 

136 specifies the maximum number of rows to buffer at a time. This is 

137 interpreted by the :class:`.BufferedRowResultProxy`, and if omitted the 

138 buffer will grow to ultimately store 1000 rows at a time.

142.. _psycopg2_executemany_mode: 

143 

144Psycopg2 Fast Execution Helpers 

145------------------------------- 

146 

147Modern versions of psycopg2 include a feature known as 

148`Fast Execution Helpers \ 

149<http://initd.org/psycopg/docs/extras.html#fast-execution-helpers>`_, which 

150have been shown in benchmarking to improve psycopg2's executemany() 

151performance, primarily with INSERT statements, by multiple orders of magnitude. 

152SQLAlchemy allows this extension to be used for all ``executemany()`` style 

153calls invoked by an :class:`.Engine` when used with :ref:`multiple parameter 

154sets <execute_multiple>`, which includes the use of this feature both by the 

155Core as well as by the ORM for inserts of objects with non-autogenerated 

156primary key values, by adding the ``executemany_mode`` flag to 

157:func:`.create_engine`:: 

158 

159 engine = create_engine( 

160 "postgresql+psycopg2://scott:tiger@host/dbname", 

161 executemany_mode='batch') 

162 

163 

164.. versionchanged:: 1.3.7 - the ``use_batch_mode`` flag has been superseded 

165 by a new parameter ``executemany_mode`` which provides support both for 

166 psycopg2's ``execute_batch`` helper as well as the ``execute_values`` 

167 helper. 

168 

169Possible options for ``executemany_mode`` include: 

170 

171* ``None`` - By default, psycopg2's extensions are not used, and the usual 

172 ``cursor.executemany()`` method is used when invoking batches of statements. 

173 

174* ``'batch'`` - Uses ``psycopg2.extras.execute_batch`` so that multiple copies 

175 of a SQL query, each one corresponding to a parameter set passed to 

176 ``executemany()``, are joined into a single SQL string separated by a 

177 semicolon. This is the same behavior as was provided by the 

178 ``use_batch_mode=True`` flag. 

179 

180* ``'values'``- For Core :func:`.insert` constructs only (including those 

181 emitted by the ORM automatically), the ``psycopg2.extras.execute_values`` 

182 extension is used so that multiple parameter sets are grouped into a single 

183 INSERT statement and joined together with multiple VALUES expressions. This 

184 method requires that the string text of the VALUES clause inside the 

185 INSERT statement is manipulated, so is only supported with a compiled 

186 :func:`.insert` construct where the format is predictable. For all other 

187 constructs, including plain textual INSERT statements not rendered by the 

188 SQLAlchemy expression language compiler, the 

189 ``psycopg2.extras.execute_batch`` method is used. It is therefore important 

190 to note that **"values" mode implies that "batch" mode is also used for 

191 all statements for which "values" mode does not apply**. 

192 

193For both strategies, the ``executemany_batch_page_size`` and 

194``executemany_values_page_size`` arguments control how many parameter sets 

195should be represented in each execution. Because "values" mode implies a 

196fallback down to "batch" mode for non-INSERT statements, there are two 

197independent page size arguments. For each, the default value of ``None`` means 

198to use psycopg2's defaults, which at the time of this writing are quite low at 

199100. For the ``execute_values`` method, a number as high as 10000 may prove 

200to be performant, whereas for ``execute_batch``, as the number represents 

201full statements repeated, a number closer to the default of 100 is likely 

202more appropriate:: 

203 

204 engine = create_engine( 

205 "postgresql+psycopg2://scott:tiger@host/dbname", 

206 executemany_mode='values', 

207 executemany_values_page_size=10000, executemany_batch_page_size=500) 

208 

209 

210.. seealso:: 

211 

212 :ref:`execute_multiple` - General information on using the 

213 :class:`.Connection` object to execute statements in such a way as to make 

214 use of the DBAPI ``.executemany()`` method. 

215 

216.. versionchanged:: 1.3.7 - Added support for 

217 ``psycopg2.extras.execute_values``. The ``use_batch_mode`` flag is 

218 superseded by the ``executemany_mode`` flag.

221.. _psycopg2_unicode:

223Unicode with Psycopg2 

224---------------------- 

225 

226By default, the psycopg2 driver uses the ``psycopg2.extensions.UNICODE`` 

227extension, such that the DBAPI receives and returns all strings as Python 

228Unicode objects directly - SQLAlchemy passes these values through without 

229change. Psycopg2 here will encode/decode string values based on the 

230current "client encoding" setting; by default this is the value in 

231the ``postgresql.conf`` file, which often defaults to ``SQL_ASCII``. 

232Typically, this can be changed to ``utf8``, as a more useful default:: 

233 

234 # postgresql.conf file 

235 

236 # client_encoding = sql_ascii # actually, defaults to database 

237 # encoding 

238 client_encoding = utf8 

239 

240A second way to affect the client encoding is to set it within Psycopg2 

241locally. SQLAlchemy will call psycopg2's 

242:meth:`psycopg2:connection.set_client_encoding` method 

243on all new connections based on the value passed to 

244:func:`.create_engine` using the ``client_encoding`` parameter:: 

245 

246 # set_client_encoding() setting; 

247 # works for *all* PostgreSQL versions 

248 engine = create_engine("postgresql://user:pass@host/dbname", 

249 client_encoding='utf8') 

250 

251This overrides the encoding specified in the PostgreSQL client configuration. 

252When using the parameter in this way, the psycopg2 driver emits 

253``SET client_encoding TO 'utf8'`` on the connection explicitly, and works 

254in all PostgreSQL versions. 

255 

256Note that the ``client_encoding`` setting as passed to :func:`.create_engine` 

257is **not the same** as the more recently added ``client_encoding`` parameter 

258now supported by libpq directly. This is enabled when ``client_encoding`` 

259is passed directly to ``psycopg2.connect()``, and from SQLAlchemy is passed 

260using the :paramref:`.create_engine.connect_args` parameter:: 

261 

262 engine = create_engine( 

263 "postgresql://user:pass@host/dbname", 

264 connect_args={'client_encoding': 'utf8'}) 

265 

266 # using the query string is equivalent 

267 engine = create_engine("postgresql://user:pass@host/dbname?client_encoding=utf8") 

268 

269The above parameter was only added to libpq as of version 9.1 of PostgreSQL, 

270so using the previous method is better for cross-version support.

272.. _psycopg2_disable_native_unicode: 

273 

274Disabling Native Unicode 

275^^^^^^^^^^^^^^^^^^^^^^^^ 

276 

277SQLAlchemy can also be instructed to skip the usage of the psycopg2 

278``UNICODE`` extension and to instead utilize its own unicode encode/decode 

279services, which are normally reserved only for those DBAPIs that don't 

280fully support unicode directly. Passing ``use_native_unicode=False`` to 

281:func:`.create_engine` will disable usage of ``psycopg2.extensions.UNICODE``. 

282SQLAlchemy will instead encode data itself into Python bytestrings on the way 

283in and coerce from bytes on the way back, 

284using the value of the :func:`.create_engine` ``encoding`` parameter, which 

285defaults to ``utf-8``. 

286SQLAlchemy's own unicode encode/decode functionality is steadily becoming 

287obsolete as most DBAPIs now support unicode fully. 

288 

289Bound Parameter Styles 

290---------------------- 

291 

292The default parameter style for the psycopg2 dialect is "pyformat", where 

293SQL is rendered using ``%(paramname)s`` style. This format has the limitation 

294that it does not accommodate the unusual case of parameter names that 

295actually contain percent or parenthesis symbols; as SQLAlchemy in many cases 

296generates bound parameter names based on the name of a column, the presence 

297of these characters in a column name can lead to problems. 

298 

299There are two solutions to the issue of a :class:`.schema.Column` that contains 

300one of these characters in its name. One is to specify the 

301:paramref:`.schema.Column.key` for columns that have such names:: 

302 

303 measurement = Table('measurement', metadata, 

304 Column('Size (meters)', Integer, key='size_meters') 

305 ) 

306 

307Above, an INSERT statement such as ``measurement.insert()`` will use 

308``size_meters`` as the parameter name, and a SQL expression such as 

309``measurement.c.size_meters > 10`` will derive the bound parameter name 

310from the ``size_meters`` key as well. 

311 

312.. versionchanged:: 1.0.0 - SQL expressions will use :attr:`.Column.key` 

313 as the source of naming when anonymous bound parameters are created 

314 in SQL expressions; previously, this behavior only applied to 

315 :meth:`.Table.insert` and :meth:`.Table.update` parameter names. 

316 

317The other solution is to use a positional format; psycopg2 allows use of the 

318"format" paramstyle, which can be passed to 

319:paramref:`.create_engine.paramstyle`:: 

320 

321 engine = create_engine( 

322 'postgresql://scott:tiger@localhost:5432/test', paramstyle='format') 

323 

324With the above engine, instead of a statement like:: 

325 

326 INSERT INTO measurement ("Size (meters)") VALUES (%(Size (meters))s) 

327 {'Size (meters)': 1} 

328 

329we instead see:: 

330 

331 INSERT INTO measurement ("Size (meters)") VALUES (%s) 

332 (1, ) 

333 

334Where above, the dictionary style is converted into a tuple with positional 

335style.

343.. _psycopg2_isolation_level:

345Psycopg2 Transaction Isolation Level 

346------------------------------------- 

347 

348As discussed in :ref:`postgresql_isolation_level`, 

349all PostgreSQL dialects support setting of transaction isolation level 

350both via the ``isolation_level`` parameter passed to :func:`.create_engine`, 

351as well as the ``isolation_level`` argument used by 

352:meth:`.Connection.execution_options`. When using the psycopg2 dialect, these 

353options make use of psycopg2's ``set_isolation_level()`` connection method, 

354rather than emitting a PostgreSQL directive; this is because psycopg2's 

355API-level setting is always emitted at the start of each transaction in any 

356case. 

357 

358The psycopg2 dialect supports these constants for isolation level: 

359 

360* ``READ COMMITTED`` 

361* ``READ UNCOMMITTED`` 

362* ``REPEATABLE READ`` 

363* ``SERIALIZABLE`` 

364* ``AUTOCOMMIT`` 

365

367 

368 :ref:`postgresql_isolation_level`

396.. _psycopg2_hstore:

398HSTORE type 

399------------ 

400 

401The ``psycopg2`` DBAPI includes an extension to natively handle marshalling of 

402the HSTORE type. The SQLAlchemy psycopg2 dialect will enable this extension 

403by default when psycopg2 version 2.4 or greater is used, and 

404it is detected that the target database has the HSTORE type set up for use. 

405In other words, when the dialect makes the first 

406connection, a sequence like the following is performed: 

407 

4081. Request the available HSTORE oids using 

409 ``psycopg2.extras.HstoreAdapter.get_oids()``. 

410 If this function returns a list of HSTORE identifiers, we then determine 

411 that the ``HSTORE`` extension is present. 

412 This function is **skipped** if the version of psycopg2 installed is 

413 less than version 2.4. 

414 

4152. If the ``use_native_hstore`` flag is at its default of ``True``, and 

416 we've detected that ``HSTORE`` oids are available, the 

417 ``psycopg2.extensions.register_hstore()`` extension is invoked for all 

418 connections. 

419 

420The ``register_hstore()`` extension has the effect of **all Python 

421dictionaries being accepted as parameters regardless of the type of target 

422column in SQL**. The dictionaries are converted by this extension into a 

423textual HSTORE expression. If this behavior is not desired, disable the 

424use of the hstore extension by setting ``use_native_hstore`` to ``False`` as 

425follows:: 

426 

427 engine = create_engine("postgresql+psycopg2://scott:tiger@localhost/test", 

428 use_native_hstore=False) 

429 

430The ``HSTORE`` type is **still supported** when the 

431``psycopg2.extensions.register_hstore()`` extension is not used. It merely 

432means that the coercion between Python dictionaries and the HSTORE 

433string format, on both the parameter side and the result side, will take 

434place within SQLAlchemy's own marshalling logic, and not that of ``psycopg2`` 

435which may be more performant.

852 def do_executemany(self, cursor, statement, parameters, context=None): 

853 if self.executemany_mode is EXECUTEMANY_DEFAULT: 

854 cursor.executemany(statement, parameters) 

855 return 

856 

857 if ( 

858 self.executemany_mode is EXECUTEMANY_VALUES 

859 and context 

860 and context.isinsert 

861 and context.compiled.insert_single_values_expr 

862 ): 

863 executemany_values = ( 

864 "(%s)" % context.compiled.insert_single_values_expr 

865 ) 

866 # guard for statement that was altered via event hook or similar 

867 if executemany_values not in statement: 

868 executemany_values = None 

869 else: 

870 executemany_values = None 

871 

872 if executemany_values: 

873 # Currently, SQLAlchemy does not pass "RETURNING" statements 

874 # into executemany(), since no DBAPI has ever supported that 

875 # until the introduction of psycopg2's executemany_values, so 

876 # we are not yet using the fetch=True flag. 

877 statement = statement.replace(executemany_values, "%s") 

878 if self.executemany_values_page_size: 

879 kwargs = {"page_size": self.executemany_values_page_size} 

880 else: 

881 kwargs = {} 

882 self._psycopg2_extras().execute_values( 

883 cursor, 

884 statement, 

885 parameters, 

886 template=executemany_values, 

887 **kwargs 

888 ) 

889 

890 else: 

891 if self.executemany_batch_page_size: 

892 kwargs = {"page_size": self.executemany_batch_page_size} 

893 else: 

894 kwargs = {} 

895 self._psycopg2_extras().execute_batch( 

896 cursor, statement, parameters, **kwargs 

897 )

908 def create_connect_args(self, url): 

909 opts = url.translate_connect_args(username="user") 

910 if opts: 

911 if "port" in opts: 

912 opts["port"] = int(opts["port"]) 

913 opts.update(url.query) 

914 # send individual dbname, user, password, host, port 

915 # parameters to psycopg2.connect() 

916 return ([], opts) 

917 elif url.query: 

918 # any other connection arguments, pass directly 

919 opts.update(url.query) 

920 return ([], opts) 

921 else: 

922 # no connection arguments whatsoever; psycopg2.connect() 

923 # requires that "dsn" be present as a blank string. 

924 return ([""], opts)

926 def is_disconnect(self, e, connection, cursor): 

927 if isinstance(e, self.dbapi.Error): 

928 # check the "closed" flag. this might not be 

929 # present on old psycopg2 versions. Also, 

930 # this flag doesn't actually help in a lot of disconnect 

931 # situations, so don't rely on it. 

932 if getattr(connection, "closed", False): 

933 return True 

934 

935 # checks based on strings. in the case that .closed 

936 # didn't cut it, fall back onto these. 

937 str_e = str(e).partition("\n")[0] 

938 for msg in [ 

939 # these error messages from libpq: interfaces/libpq/fe-misc.c 

940 # and interfaces/libpq/fe-secure.c. 

941 "terminating connection", 

942 "closed the connection", 

943 "connection not open", 

944 "could not receive data from server", 

945 "could not send data to server", 

946 # psycopg2 client errors, psycopg2/conenction.h, 

947 # psycopg2/cursor.h 

948 "connection already closed", 

949 "cursor already closed", 

950 # not sure where this path is originally from, it may 

951 # be obsolete. It really says "losed", not "closed". 

952 "losed the connection unexpectedly", 

953 # these can occur in newer SSL 

954 "connection has been closed unexpectedly", 

955 "SSL SYSCALL error: Bad file descriptor", 

956 "SSL SYSCALL error: EOF detected", 

957 "SSL error: decryption failed or bad record mac", 

958 "SSL SYSCALL error: Operation timed out", 

959 ]: 

960 idx = str_e.find(msg) 

961 if idx >= 0 and '"' not in str_e[:idx]: 

962 return True 

963 return False
