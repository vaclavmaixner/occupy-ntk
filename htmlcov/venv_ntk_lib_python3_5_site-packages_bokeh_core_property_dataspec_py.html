76class DataSpec(Either): 

77 ''' Base class for properties that accept either a fixed value, or a 

78 string name that references a column in a 

79 :class:`~bokeh.models.sources.ColumnDataSource`. 

80 

81 Many Bokeh models have properties that a user might want to set either 

82 to a single fixed value, or to have the property take values from some 

83 column in a data source. As a concrete example consider a glyph with 

84 an ``x`` property for location. We might want to set all the glyphs 

85 that get drawn to have the same location, say ``x=10``. It would be 

86 convenient to just be able to write: 

87 

88 .. code-block:: python 

89 

90 glyph.x = 10 

91 

92 Alternatively, maybe the each glyph that gets drawn should have a 

93 different location, according to the "pressure" column of a data 

94 source. In this case we would like to be able to write: 

95 

96 .. code-block:: python 

97 

98 glyph.x = "pressure" 

99 

100 Bokeh ``DataSpec`` properties (and subclasses) afford this ease of 

101 and consistency of expression. Ultimately, all ``DataSpec`` properties 

102 resolve to dictionary values, with either a ``"value"`` key, or a 

103 ``"field"`` key, depending on how it is set. 

104 

105 For instance: 

106 

107 .. code-block:: python 

108 

109 glyph.x = 10 # => { 'value': 10 } 

110 

111 glyph.x = "pressure" # => { 'field': 'pressure' } 

112 

113 When these underlying dictionary dictionary values are received in 

114 the browser, BokehJS knows how to interpret them and take the correct, 

115 expected action (i.e., draw the glyph at ``x=10``, or draw the glyph 

116 with ``x`` coordinates from the "pressure" column). In this way, both 

117 use-cases may be expressed easily in python, without having to handle 

118 anything differently, from the user perspective. 

119 

120 It is worth noting that ``DataSpec`` properties can also be set directly 

121 with properly formed dictionary values: 

122 

123 .. code-block:: python 

124 

125 glyph.x = { 'value': 10 } # same as glyph.x = 10 

126 

127 glyph.x = { 'field': 'pressure' } # same as glyph.x = "pressure" 

128 

129 Setting the property directly as a dict can be useful in certain 

130 situations. For instance some ``DataSpec`` subclasses also add a 

131 ``"units"`` key to the dictionary. This key is often set automatically, 

132 but the dictionary format provides a direct mechanism to override as 

133 necessary. Additionally, ``DataSpec`` can have a ``"transform"`` key, 

134 that specifies a client-side transform that should be applied to any 

135 fixed or field values before they are uses. As an example, you might want 

136 to apply a ``Jitter`` transform to the ``x`` values: 

137 

138 .. code-block:: python 

139 

140 glyph.x = { 'value': 10, 'transform': Jitter(width=0.4) } 

141 

142 Note that ``DataSpec`` is not normally useful on its own. Typically, 

143 a model will define properties using one of the subclasses such 

144 as :class:`~bokeh.core.properties.NumberSpec` or 

145 :class:`~bokeh.core.properties.ColorSpec`. For example, a Bokeh 

146 model with ``x``, ``y`` and ``color`` properties that can handle 

147 fixed values or columns automatically might look like: 

148 

149 .. code-block:: python 

150 

151 class SomeModel(Model): 

152 

153 x = NumberSpec(default=0, help="docs for x") 

154 

155 y = NumberSpec(default=0, help="docs for y") 

156 

157 color = ColorSpec(help="docs for color") # defaults to None 

158 

159 '''

524class ColorSpec(DataSpec): 

525 ''' A |DataSpec| property that accepts |Color| fixed values. 

526 

527 The ``ColorSpec`` property attempts to first interpret string values as 

528 colors. Otherwise, string values are interpreted as field names. For 

529 example: 

530 

531 .. code-block:: python 

532 

533 m.color = "#a4225f" # value (hex color string) 

534 

535 m.color = "firebrick" # value (named CSS color string) 

536 

537 m.color = "foo" # field (named "foo") 

538 

539 This automatic interpretation can be override using the dict format 

540 directly, or by using the |field| function: 

541 

542 .. code-block:: python 

543 

544 m.color = { "field": "firebrick" } # field (named "firebrick") 

545 

546 m.color = field("firebrick") # field (named "firebrick") 

547 

548 ''' 

549 def __init__(self, default, help=None, key_type=_ExprFieldValueTransform): 

550 super(ColorSpec, self).__init__(key_type, Color, default=default, help=help) 

551 

552 @classmethod 

553 def isconst(cls, val): 

554 ''' Whether the value is a string color literal. 

555 

556 Checks for a well-formed hexadecimal color value or a named color. 

557 

558 Args: 

559 val (str) : the value to check 

560 

561 Returns: 

562 True, if the value is a string color literal 

563 

564 ''' 

565 return isinstance(val, string_types) and \ 

566 ((len(val) == 7 and val[0] == "#") or val in enums.NamedColor) 

567 

568 def to_serializable(self, obj, name, val): 

569 if val is None: 

570 return dict(value=None) 

571 

572 # Check for hexadecimal or named color 

573 if self.isconst(val): 

574 return dict(value=val) 

575 

576 # Check for RGB or RGBa tuple 

577 if isinstance(val, tuple): 

578 return dict(value=colors.RGB(*val).to_css()) 

579 

580 # Check for data source field name 

581 if isinstance(val, colors.RGB): 

582 return val.to_css() 

583 

584 # Check for data source field name or rgb(a) string 

585 if isinstance(val, string_types): 

586 if val.startswith(("rgb(", "rgba(")): 

587 return val 

588 

589 return dict(field=val) 

590 

591 # Must be dict, return new dict 

592 return dict(val)

594 def prepare_value(self, cls, name, value): 

595 # Some explanation is in order. We want to accept tuples like 

596 # (12.0, 100.0, 52.0) i.e. that have "float" byte values. The 

597 # ColorSpec has a transform to adapt values like this to tuples 

598 # of integers, but Property validation happens before the 

599 # transform step, so values like that will fail Color validation 

600 # at this point, since Color is very strict about only accepting 

601 # tuples of (integer) bytes. This conditions tuple values to only 

602 # have integer RGB components 

603 if isinstance(value, tuple): 

604 # TODO (bev) verify that all original floats are integer values? 

605 value = tuple(int(v) if i < 3 else v for i, v in enumerate(value)) 

606 return super(ColorSpec, self).prepare_value(cls, name, value)

655def value(val, transform=None): 

656 ''' Convenience function to explicitly return a "value" specification for 

657 a Bokeh :class:`~bokeh.core.properties.DataSpec` property. 

658 

659 Args: 

660 val (any) : a fixed value to specify for a ``DataSpec`` property. 

661 

662 transform (Transform, optional) : a transform to apply (default: None) 

663 

664 Returns: 

665 dict : ``{ "value": name }`` 

666 

667 .. note:: 

668 String values for property specifications are by default interpreted 

669 as field names. This function is especially useful when you want to 

670 specify a fixed value with text properties. 

671 

672 Example: 

673 

674 .. code-block:: python 

675 

676 # The following will take text values to render from a data source
