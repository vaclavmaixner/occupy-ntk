97@array_function_dispatch(_take_dispatcher) 

98def take(a, indices, axis=None, out=None, mode='raise'): 

99 """ 

100 Take elements from an array along an axis. 

101 

102 When axis is not None, this function does the same thing as "fancy" 

103 indexing (indexing arrays using arrays); however, it can be easier to use 

104 if you need elements along a given axis. A call such as 

105 ``np.take(arr, indices, axis=3)`` is equivalent to 

106 ``arr[:,:,:,indices,...]``. 

107 

108 Explained without fancy indexing, this is equivalent to the following use 

109 of `ndindex`, which sets each of ``ii``, ``jj``, and ``kk`` to a tuple of 

110 indices:: 

111 

112 Ni, Nk = a.shape[:axis], a.shape[axis+1:] 

113 Nj = indices.shape 

114 for ii in ndindex(Ni): 

115 for jj in ndindex(Nj): 

116 for kk in ndindex(Nk): 

117 out[ii + jj + kk] = a[ii + (indices[jj],) + kk] 

118 

119 Parameters 

120 ---------- 

121 a : array_like (Ni..., M, Nk...) 

122 The source array. 

123 indices : array_like (Nj...) 

124 The indices of the values to extract. 

125 

126 .. versionadded:: 1.8.0 

127 

128 Also allow scalars for indices. 

129 axis : int, optional 

130 The axis over which to select values. By default, the flattened 

131 input array is used. 

132 out : ndarray, optional (Ni..., Nj..., Nk...) 

133 If provided, the result will be placed in this array. It should 

134 be of the appropriate shape and dtype. Note that `out` is always 

135 buffered if `mode='raise'`; use other modes for better performance. 

136 mode : {'raise', 'wrap', 'clip'}, optional 

137 Specifies how out-of-bounds indices will behave. 

138 

139 * 'raise' -- raise an error (default) 

140 * 'wrap' -- wrap around 

141 * 'clip' -- clip to the range 

142 

143 'clip' mode means that all indices that are too large are replaced 

144 by the index that addresses the last element along that axis. Note 

145 that this disables indexing with negative numbers. 

146 

147 Returns 

148 ------- 

149 out : ndarray (Ni..., Nj..., Nk...) 

150 The returned array has the same type as `a`. 

151 

152 See Also 

153 -------- 

154 compress : Take elements using a boolean mask 

155 ndarray.take : equivalent method 

156 take_along_axis : Take elements by matching the array and the index arrays 

157 

158 Notes 

159 ----- 

160 

161 By eliminating the inner loop in the description above, and using `s_` to 

162 build simple slice objects, `take` can be expressed in terms of applying 

163 fancy indexing to each 1-d slice:: 

164 

165 Ni, Nk = a.shape[:axis], a.shape[axis+1:] 

166 for ii in ndindex(Ni): 

167 for kk in ndindex(Nj): 

168 out[ii + s_[...,] + kk] = a[ii + s_[:,] + kk][indices] 

169 

170 For this reason, it is equivalent to (but faster than) the following use 

171 of `apply_along_axis`:: 

172 

173 out = np.apply_along_axis(lambda a_1d: a_1d[indices], axis, a) 

174 

175 Examples 

176 -------- 

177 >>> a = [4, 3, 5, 7, 6, 8] 

178 >>> indices = [0, 1, 4] 

179 >>> np.take(a, indices) 

180 array([4, 3, 6]) 

181 

182 In this example if `a` is an ndarray, "fancy" indexing can be used. 

183 

184 >>> a = np.array(a) 

185 >>> a[indices] 

186 array([4, 3, 6]) 

187 

188 If `indices` is not one dimensional, the output also has these dimensions. 

189 

190 >>> np.take(a, [[0, 1], [2, 3]]) 

191 array([[4, 3], 

192 [5, 7]]) 

193 """ 

194 return _wrapfunc(a, 'take', indices, axis=axis, out=out, mode=mode)

282 array([[0, 4, 3],

294 array([1, 4, 2, 5, 3, 6])

311@array_function_dispatch(_choose_dispatcher) 

312def choose(a, choices, out=None, mode='raise'): 

313 """ 

314 Construct an array from an index array and a set of arrays to choose from. 

315 

316 First of all, if confused or uncertain, definitely look at the Examples - 

317 in its full generality, this function is less simple than it might 

318 seem from the following code description (below ndi = 

319 `numpy.lib.index_tricks`): 

320 

321 ``np.choose(a,c) == np.array([c[a[I]][I] for I in ndi.ndindex(a.shape)])``. 

322 

323 But this omits some subtleties. Here is a fully general summary: 

324 

325 Given an "index" array (`a`) of integers and a sequence of `n` arrays 

326 (`choices`), `a` and each choice array are first broadcast, as necessary, 

327 to arrays of a common shape; calling these *Ba* and *Bchoices[i], i = 

328 0,...,n-1* we have that, necessarily, ``Ba.shape == Bchoices[i].shape`` 

329 for each `i`. Then, a new array with shape ``Ba.shape`` is created as 

330 follows: 

331 

332 * if ``mode=raise`` (the default), then, first of all, each element of 

333 `a` (and thus `Ba`) must be in the range `[0, n-1]`; now, suppose that 

334 `i` (in that range) is the value at the `(j0, j1, ..., jm)` position 

335 in `Ba` - then the value at the same position in the new array is the 

336 value in `Bchoices[i]` at that same position; 

337 

338 * if ``mode=wrap``, values in `a` (and thus `Ba`) may be any (signed) 

339 integer; modular arithmetic is used to map integers outside the range 

340 `[0, n-1]` back into that range; and then the new array is constructed 

341 as above; 

342 

343 * if ``mode=clip``, values in `a` (and thus `Ba`) may be any (signed) 

344 integer; negative integers are mapped to 0; values greater than `n-1` 

345 are mapped to `n-1`; and then the new array is constructed as above. 

346 

347 Parameters 

348 ---------- 

349 a : int array 

350 This array must contain integers in `[0, n-1]`, where `n` is the number 

351 of choices, unless ``mode=wrap`` or ``mode=clip``, in which cases any 

352 integers are permissible. 

353 choices : sequence of arrays 

354 Choice arrays. `a` and all of the choices must be broadcastable to the 

355 same shape. If `choices` is itself an array (not recommended), then 

356 its outermost dimension (i.e., the one corresponding to 

357 ``choices.shape[0]``) is taken as defining the "sequence". 

358 out : array, optional 

359 If provided, the result will be inserted into this array. It should 

360 be of the appropriate shape and dtype. Note that `out` is always 

361 buffered if `mode='raise'`; use other modes for better performance. 

362 mode : {'raise' (default), 'wrap', 'clip'}, optional 

363 Specifies how indices outside `[0, n-1]` will be treated: 

364 

365 * 'raise' : an exception is raised 

366 * 'wrap' : value becomes value mod `n` 

367 * 'clip' : values < 0 are mapped to 0, values > n-1 are mapped to n-1 

368 

369 Returns 

370 ------- 

371 merged_array : array 

372 The merged result. 

373 

374 Raises 

375 ------ 

376 ValueError: shape mismatch 

377 If `a` and each choice array are not all broadcastable to the same 

378 shape. 

379 

380 See Also 

381 -------- 

382 ndarray.choose : equivalent method 

383 numpy.take_along_axis : Preferable if `choices` is an array 

384 

385 Notes 

386 ----- 

387 To reduce the chance of misinterpretation, even though the following 

388 "abuse" is nominally supported, `choices` should neither be, nor be 

389 thought of as, a single array, i.e., the outermost sequence-like container 

390 should be either a list or a tuple. 

391 

392 Examples 

393 -------- 

394 

395 >>> choices = [[0, 1, 2, 3], [10, 11, 12, 13], 

396 ... [20, 21, 22, 23], [30, 31, 32, 33]] 

397 >>> np.choose([2, 3, 1, 0], choices 

398 ... # the first element of the result will be the first element of the 

399 ... # third (2+1) "array" in choices, namely, 20; the second element 

400 ... # will be the second element of the fourth (3+1) choice array, i.e., 

401 ... # 31, etc. 

402 ... ) 

403 array([20, 31, 12, 3]) 

404 >>> np.choose([2, 4, 1, 0], choices, mode='clip') # 4 goes to 3 (4-1) 

405 array([20, 31, 12, 3]) 

406 >>> # because there are 4 choice arrays 

407 >>> np.choose([2, 4, 1, 0], choices, mode='wrap') # 4 goes to (4 mod 4) 

408 array([20, 1, 12, 3]) 

409 >>> # i.e., 0 

410 

411 A couple examples illustrating how choose broadcasts: 

412 

413 >>> a = [[1, 0, 1], [0, 1, 0], [1, 0, 1]] 

414 >>> choices = [-10, 10] 

415 >>> np.choose(a, choices) 

416 array([[ 10, -10, 10], 

417 [-10, 10, -10], 

418 [ 10, -10, 10]]) 

419 

420 >>> # With thanks to Anne Archibald 

421 >>> a = np.array([0, 1]).reshape((2,1,1)) 

422 >>> c1 = np.array([1, 2, 3]).reshape((1,3,1)) 

423 >>> c2 = np.array([-1, -2, -3, -4, -5]).reshape((1,1,5)) 

424 >>> np.choose(a, (c1, c2)) # result is 2x3x5, res[0,:,:]=c1, res[1,:,:]=c2 

425 array([[[ 1, 1, 1, 1, 1], 

426 [ 2, 2, 2, 2, 2], 

427 [ 3, 3, 3, 3, 3]], 

428 [[-1, -2, -3, -4, -5], 

429 [-1, -2, -3, -4, -5], 

430 [-1, -2, -3, -4, -5]]]) 

431 

432 """ 

433 return _wrapfunc(a, 'choose', choices, out=out, mode=mode)

659def partition(a, kth, axis=-1, kind='introselect', order=None): 

660 """ 

661 Return a partitioned copy of an array. 

662 

663 Creates a copy of the array with its elements rearranged in such a 

664 way that the value of the element in k-th position is in the

666 the k-th element are moved before this element and all equal or

851 and 'mergesort' use timsort or radix sort under the covers and, in general,

855 .. versionchanged:: 1.15.0.

864 

865 Returns

876 partition : Partial sort. 

877 

878 Notes 

879 ----- 

880 The various sorting algorithms are characterized by their average speed, 

881 worst case performance, work space size, and whether they are stable. A 

882 stable sort keeps items with the same key in the same relative 

883 order. The four algorithms implemented in NumPy have the following 

884 properties: 

885 

886 =========== ======= ============= ============ ======== 

887 kind speed worst case work space stable 

888 =========== ======= ============= ============ ======== 

889 'quicksort' 1 O(n^2) 0 no 

890 'heapsort' 3 O(n*log(n)) 0 no 

891 'mergesort' 2 O(n*log(n)) ~n/2 yes 

892 'timsort' 2 O(n*log(n)) ~n/2 yes 

893 =========== ======= ============= ============ ======== 

894 

895 .. note:: The datatype determines which of 'mergesort' or 'timsort' 

896 is actually used, even if 'mergesort' is specified. User selection 

897 at a finer scale is not currently available. 

898 

899 All the sort algorithms make temporary copies of the data when 

900 sorting along any but the last axis. Consequently, sorting along 

901 the last axis is faster and uses less space than sorting along 

902 any other axis. 

903 

904 The sort order for complex numbers is lexicographic. If both the real 

905 and imaginary parts are non-nan then the order is determined by the 

906 real parts except when they are equal, in which case the order is 

907 determined by the imaginary parts. 

908 

909 Previous to numpy 1.4.0 sorting real and complex arrays containing nan 

910 values led to undefined behaviour. In numpy versions >= 1.4.0 nan 

911 values are sorted to the end. The extended sort order is: 

912 

913 * Real: [R, nan] 

914 * Complex: [R + Rj, R + nanj, nan + Rj, nan + nanj] 

915 

916 where R is a non-nan real value. Complex values with the same nan 

917 placements are sorted according to the non-nan part if it exists. 

918 Non-nan values are sorted as before. 

919 

920 .. versionadded:: 1.12.0 

921 

922 quicksort has been changed to `introsort <https://en.wikipedia.org/wiki/Introsort>`_. 

923 When sorting does not make enough progress it switches to 

924 `heapsort <https://en.wikipedia.org/wiki/Heapsort>`_. 

925 This implementation makes quicksort O(n*log(n)) in the worst case. 

926 

927 'stable' automatically chooses the best stable sorting algorithm 

928 for the data type being sorted. 

929 It, along with 'mergesort' is currently mapped to 

930 `timsort <https://en.wikipedia.org/wiki/Timsort>`_ 

931 or `radix sort <https://en.wikipedia.org/wiki/Radix_sort>`_ 

932 depending on the data type. 

933 API forward compatibility currently limits the 

934 ability to select the implementation and it is hardwired for the different 

935 data types. 

936 

937 .. versionadded:: 1.17.0 

938 

939 Timsort is added for better performance on already or nearly 

940 sorted data. On random data timsort is almost identical to 

941 mergesort. It is now used for stable sort while quicksort is still the 

942 default sort if none is chosen. For timsort details, refer to 

943 `CPython listsort.txt <https://github.com/python/cpython/blob/3.7/Objects/listsort.txt>`_. 

944 'mergesort' and 'stable' are mapped to radix sort for integer data types. Radix sort is an 

945 O(n) sort instead of O(n log n). 

946 

947 .. versionchanged:: 1.17.0 

948 

949 NaT now sorts to the end of arrays for consistency with NaN.

974 

975 Sort by age, then height if ages are equal: 

976 

977 >>> np.sort(a, order=['age', 'height']) # doctest: +SKIP

980 dtype=[('name', '|S10'), ('height', '<f8'), ('age', '<i4')])

1694def ravel(a, order='C'): 

1695 """Return a contiguous flattened array. 

1696

1698 made only if needed.

1710

1712 to index the elements in row-major, C-style order, 

1713 with the last axis index changing fastest, back to the first 

1714 axis index changing slowest. 'F' means to index the elements 

1715 in column-major, Fortran-style order, with the 

1716 first index changing fastest, and the last index changing 

1717 slowest. Note that the 'C' and 'F' options take no account of 

1718 the memory layout of the underlying array, and only refer to 

1719 the order of axis indexing. 'A' means to read the elements in 

1720 Fortran-like index order if `a` is Fortran *contiguous* in 

1721 memory, C-like order otherwise. 'K' means to read the 

1722 elements in the order they occur in memory, except for 

1723 reversing the data when strides are negative. By default, 'C' 

1724 index order is used. 

1725 

1726 Returns 

1727 -------

1744 be generalized to multiple dimensions, where row-major order 

1745 implies that the index along the first axis varies slowest, and 

1746 the index along the last quickest. The opposite holds for

3167 -----

3169 rounds to the nearest even value. Thus 1.5 and 2.5 round to 2.0, 

3170 -0.5 and 0.5 round to 0.0, etc. 

3171 

3172 ``np.around`` uses a fast but sometimes inexact algorithm to round 

3173 floating-point datatypes. For positive `decimals` it is equivalent to 

3174 ``np.true_divide(np.rint(a * 10**decimals), 10**decimals)``, which has 

3175 error due to the inexact representation of decimal fractions in the IEEE 

3176 floating point standard [1]_ and errors introduced when scaling by powers 

3177 of ten. For instance, note the extra "1" in the following: 

3178 

3179 >>> np.round(56294995342131.5, 3) 

3180 56294995342131.51

3183 preferable to use numpy's float printing routines to limit the number of 

3184 printed decimals: 

3185 

3186 >>> np.format_float_positional(56294995342131.5, precision=3) 

3187 '56294995342131.5' 

3188 

3189 The float printing routines use an accurate but much more computationally 

3190 demanding algorithm to compute the number of digits after the decimal 

3191 point. 

3192

3194 but slower algorithm for 64-bit floating point values: 

3195 

3196 >>> round(56294995342131.5, 3)

3198 >>> np.round(16.055, 2), round(16.055, 2) # equals 16.0549999999999997

3205 https://people.eecs.berkeley.edu/~wkahan/ieee754status/IEEE754.PDF 

3206 .. [2] "How Futile are Mindless Assessments of 

3207 Roundoff in Floating-Point Computation?", William Kahan, 

3208 https://people.eecs.berkeley.edu/~wkahan/Mindless.pdf
