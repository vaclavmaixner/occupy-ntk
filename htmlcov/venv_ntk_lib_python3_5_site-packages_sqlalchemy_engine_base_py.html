28class Connection(Connectable): 

29 """Provides high-level functionality for a wrapped DB-API connection. 

30 

31 Provides execution support for string-based SQL statements as well as 

32 :class:`.ClauseElement`, :class:`.Compiled` and :class:`.DefaultGenerator` 

33 objects. Provides a :meth:`begin` method to return :class:`.Transaction` 

34 objects. 

35 

36 The Connection object is **not** thread-safe. While a Connection can be 

37 shared among threads using properly synchronized access, it is still 

38 possible that the underlying DBAPI connection may not support shared 

39 access between threads. Check the DBAPI documentation for details. 

40

42 from the connection pool. In this state, the connection pool has no affect 

43 upon the connection, including its expiration or timeout state. For the 

44 connection pool to properly manage connections, connections should be 

45 returned to the connection pool (i.e. ``connection.close()``) whenever the 

46 connection is not in use. 

47

183 def execution_options(self, **opt): 

184 r""" Set non-SQL options for the connection which take effect 

185 during execution. 

186 

187 The method returns a copy of this :class:`.Connection` which references 

188 the same underlying DBAPI connection, but also defines the given 

189 execution options which will take effect for a call to 

190 :meth:`execute`. As the new :class:`.Connection` references the same 

191 underlying resource, it's usually a good idea to ensure that the copies 

192 will be discarded immediately, which is implicit if used as in:: 

193 

194 result = connection.execution_options(stream_results=True).\ 

195 execute(stmt) 

196 

197 Note that any key/value can be passed to 

198 :meth:`.Connection.execution_options`, and it will be stored in the 

199 ``_execution_options`` dictionary of the :class:`.Connection`. It 

200 is suitable for usage by end-user schemes to communicate with 

201 event listeners, for example. 

202 

203 The keywords that are currently recognized by SQLAlchemy itself 

204 include all those listed under :meth:`.Executable.execution_options`, 

205 as well as others that are specific to :class:`.Connection`. 

206 

207 :param autocommit: Available on: Connection, statement. 

208 When True, a COMMIT will be invoked after execution 

209 when executed in 'autocommit' mode, i.e. when an explicit 

210 transaction is not begun on the connection. Note that DBAPI 

211 connections by default are always in a transaction - SQLAlchemy uses 

212 rules applied to different kinds of statements to determine if 

213 COMMIT will be invoked in order to provide its "autocommit" feature. 

214 Typically, all INSERT/UPDATE/DELETE statements as well as 

215 CREATE/DROP statements have autocommit behavior enabled; SELECT 

216 constructs do not. Use this option when invoking a SELECT or other 

217 specific SQL construct where COMMIT is desired (typically when 

218 calling stored procedures and such), and an explicit 

219 transaction is not in progress. 

220 

221 :param compiled_cache: Available on: Connection. 

222 A dictionary where :class:`.Compiled` objects 

223 will be cached when the :class:`.Connection` compiles a clause 

224 expression into a :class:`.Compiled` object. 

225 It is the user's responsibility to 

226 manage the size of this dictionary, which will have keys 

227 corresponding to the dialect, clause element, the column 

228 names within the VALUES or SET clause of an INSERT or UPDATE, 

229 as well as the "batch" mode for an INSERT or UPDATE statement. 

230 The format of this dictionary is not guaranteed to stay the 

231 same in future releases. 

232 

233 Note that the ORM makes use of its own "compiled" caches for 

234 some operations, including flush operations. The caching 

235 used by the ORM internally supersedes a cache dictionary 

236 specified here. 

237 

238 :param isolation_level: Available on: :class:`.Connection`. 

239 

240 Set the transaction isolation level for the lifespan of this 

241 :class:`.Connection` object. Valid values include those string 

242 values accepted by the :paramref:`.create_engine.isolation_level` 

243 parameter passed to :func:`.create_engine`. These levels are 

244 semi-database specific; see individual dialect documentation for 

245 valid levels. 

246 

247 The isolation level option applies the isolation level by emitting 

248 statements on the DBAPI connection, and **necessarily affects the 

249 original Connection object overall**, not just the copy that is 

250 returned by the call to :meth:`.Connection.execution_options` 

251 method. The isolation level will remain at the given setting until 

252 the DBAPI connection itself is returned to the connection pool, i.e. 

253 the :meth:`.Connection.close` method on the original 

254 :class:`.Connection` is called, where an event handler will emit 

255 additional statements on the DBAPI connection in order to revert the 

256 isolation level change. 

257 

258 .. warning:: The ``isolation_level`` execution option should 

259 **not** be used when a transaction is already established, that 

260 is, the :meth:`.Connection.begin` method or similar has been 

261 called. A database cannot change the isolation level on a 

262 transaction in progress, and different DBAPIs and/or 

263 SQLAlchemy dialects may implicitly roll back or commit 

264 the transaction, or not affect the connection at all. 

265 

266 .. note:: The ``isolation_level`` execution option is implicitly 

267 reset if the :class:`.Connection` is invalidated, e.g. via 

268 the :meth:`.Connection.invalidate` method, or if a 

269 disconnection error occurs. The new connection produced after 

270 the invalidation will not have the isolation level re-applied 

271 to it automatically. 

272

290 :param no_parameters: When ``True``, if the final parameter 

291 list or dictionary is totally empty, will invoke the 

292 statement on the cursor as ``cursor.execute(statement)``, 

293 not passing the parameter collection at all. 

294 Some DBAPIs such as psycopg2 and mysql-python consider 

295 percent signs as significant only when parameters are 

296 present; this option allows code to generate SQL 

297 containing percent signs (and possibly other characters) 

298 that is neutral regarding whether it's executed by the DBAPI

302 :param stream_results: Available on: Connection, statement.

304 "streamed" and not pre-buffered, if possible. This is a limitation 

305 of many DBAPIs. The flag is currently understood only by the
