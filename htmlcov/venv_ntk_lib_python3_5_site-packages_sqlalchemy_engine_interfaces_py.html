15class Dialect(object): 

16 """Define the behavior of a specific database and DB-API combination. 

17 

18 Any aspect of metadata definition, SQL query generation, 

19 execution, result-set handling, or anything else which varies 

20 between databases is defined under the general category of the 

21 Dialect. The Dialect acts as a factory for other 

22 database-specific object implementations including 

23 ExecutionContext, Compiled, DefaultGenerator, and TypeEngine. 

24 

25 All Dialects implement the following attributes: 

26 

27 name 

28 identifying name for the dialect from a DBAPI-neutral point of view 

29 (i.e. 'sqlite') 

30 

31 driver 

32 identifying name for the dialect's DBAPI 

33 

34 positional 

35 True if the paramstyle for this Dialect is positional. 

36 

37 paramstyle 

38 the paramstyle to be used (some DB-APIs support multiple 

39 paramstyles). 

40 

41 convert_unicode 

42 True if Unicode conversion should be applied to all ``str`` 

43 types. 

44 

45 encoding 

46 type of encoding to use for unicode, usually defaults to 

47 'utf-8'. 

48 

49 statement_compiler 

50 a :class:`.Compiled` class used to compile SQL statements 

51 

52 ddl_compiler 

53 a :class:`.Compiled` class used to compile DDL statements 

54 

55 server_version_info 

56 a tuple containing a version number for the DB backend in use. 

57 This value is only available for supporting dialects, and is 

58 typically populated during the initial connection to the database. 

59 

60 default_schema_name 

61 the name of the default schema. This value is only available for 

62 supporting dialects, and is typically populated during the 

63 initial connection to the database. 

64 

65 execution_ctx_cls 

66 a :class:`.ExecutionContext` class used to handle statement execution 

67 

68 execute_sequence_format 

69 either the 'tuple' or 'list' type, depending on what cursor.execute() 

70 accepts for the second argument (they vary). 

71 

72 preparer 

73 a :class:`~sqlalchemy.sql.compiler.IdentifierPreparer` class used to 

74 quote identifiers. 

75 

76 supports_alter 

77 ``True`` if the database supports ``ALTER TABLE``. 

78 

79 max_identifier_length 

80 The maximum length of identifier names. 

81 

82 supports_unicode_statements 

83 Indicate whether the DB-API can receive SQL statements as Python 

84 unicode strings 

85 

86 supports_unicode_binds 

87 Indicate whether the DB-API can receive string bind parameters 

88 as Python unicode strings 

89 

90 supports_sane_rowcount 

91 Indicate whether the dialect properly implements rowcount for 

92 ``UPDATE`` and ``DELETE`` statements. 

93 

94 supports_sane_multi_rowcount 

95 Indicate whether the dialect properly implements rowcount for 

96 ``UPDATE`` and ``DELETE`` statements when executed via 

97 executemany. 

98 

99 preexecute_autoincrement_sequences 

100 True if 'implicit' primary key functions must be executed separately 

101 in order to get their value. This is currently oriented towards 

102 PostgreSQL. 

103 

104 implicit_returning 

105 use RETURNING or equivalent during INSERT execution in order to load 

106 newly generated primary keys and other column defaults in one execution, 

107 which are then available via inserted_primary_key. 

108 If an insert statement has returning() specified explicitly, 

109 the "implicit" functionality is not used and inserted_primary_key 

110 will not be available. 

111 

112 colspecs 

113 A dictionary of TypeEngine classes from sqlalchemy.types mapped 

114 to subclasses that are specific to the dialect class. This 

115 dictionary is class-level only and is not accessed from the 

116 dialect instance itself. 

117 

118 supports_default_values 

119 Indicates if the construct ``INSERT INTO tablename DEFAULT 

120 VALUES`` is supported 

121 

122 supports_sequences 

123 Indicates if the dialect supports CREATE SEQUENCE or similar. 

124 

125 sequences_optional 

126 If True, indicates if the "optional" flag on the Sequence() construct 

127 should signal to not generate a CREATE SEQUENCE. Applies only to 

128 dialects that support sequences. Currently used only to allow PostgreSQL 

129 SERIAL to be used on a column that specifies Sequence() for usage on 

130 other backends. 

131 

132 supports_native_enum 

133 Indicates if the dialect supports a native ENUM construct. 

134 This will prevent types.Enum from generating a CHECK 

135 constraint when that type is used. 

136 

137 supports_native_boolean 

138 Indicates if the dialect supports a native boolean construct. 

139 This will prevent types.Boolean from generating a CHECK 

140 constraint when that type is used. 

141 

142 dbapi_exception_translation_map 

143 A dictionary of names that will contain as values the names of 

144 pep-249 exceptions ("IntegrityError", "OperationalError", etc) 

145 keyed to alternate class names, to support the case where a 

146 DBAPI has exception classes that aren't named as they are 

147 referred to (e.g. IntegrityError = MyException). In the vast 

148 majority of cases this dictionary is empty. 

149 

150 .. versionadded:: 1.0.5 

151 

152 """

816class CreateEnginePlugin(object): 

817 """A set of hooks intended to augment the construction of an 

818 :class:`.Engine` object based on entrypoint names in a URL. 

819 

820 The purpose of :class:`.CreateEnginePlugin` is to allow third-party 

821 systems to apply engine, pool and dialect level event listeners without 

822 the need for the target application to be modified; instead, the plugin 

823 names can be added to the database URL. Target applications for 

824 :class:`.CreateEnginePlugin` include: 

825 

826 * connection and SQL performance tools, e.g. which use events to track 

827 number of checkouts and/or time spent with statements 

828 

829 * connectivity plugins such as proxies 

830 

831 Plugins are registered using entry points in a similar way as that 

832 of dialects:: 

833 

834 entry_points={ 

835 'sqlalchemy.plugins': [ 

836 'myplugin = myapp.plugins:MyPlugin' 

837 ] 

838 

839 A plugin that uses the above names would be invoked from a database 

840 URL as in:: 

841 

842 from sqlalchemy import create_engine 

843 

844 engine = create_engine( 

845 "mysql+pymysql://scott:tiger@localhost/test?plugin=myplugin") 

846 

847 Alternatively, the :paramref:`.create_engine.plugins" argument may be 

848 passed as a list to :func:`.create_engine`:: 

849 

850 engine = create_engine( 

851 "mysql+pymysql://scott:tiger@localhost/test", 

852 plugins=["myplugin"]) 

853 

854 .. versionadded:: 1.2.3 plugin names can also be specified 

855 to :func:`.create_engine` as a list 

856 

857 The ``plugin`` argument supports multiple instances, so that a URL 

858 may specify multiple plugins; they are loaded in the order stated 

859 in the URL:: 

860 

861 engine = create_engine( 

862 "mysql+pymysql://scott:tiger@localhost/" 

863 "test?plugin=plugin_one&plugin=plugin_twp&plugin=plugin_three") 

864 

865 A plugin can receive additional arguments from the URL string as 

866 well as from the keyword arguments passed to :func:`.create_engine`. 

867 The :class:`.URL` object and the keyword dictionary are passed to the 

868 constructor so that these arguments can be extracted from the url's 

869 :attr:`.URL.query` collection as well as from the dictionary:: 

870 

871 class MyPlugin(CreateEnginePlugin): 

872 def __init__(self, url, kwargs): 

873 self.my_argument_one = url.query.pop('my_argument_one') 

874 self.my_argument_two = url.query.pop('my_argument_two') 

875 self.my_argument_three = kwargs.pop('my_argument_three', None) 

876 

877 Arguments like those illustrated above would be consumed from the 

878 following:: 

879 

880 from sqlalchemy import create_engine 

881 

882 engine = create_engine( 

883 "mysql+pymysql://scott:tiger@localhost/" 

884 "test?plugin=myplugin&my_argument_one=foo&my_argument_two=bar", 

885 my_argument_three='bat') 

886 

887 The URL and dictionary are used for subsequent setup of the engine 

888 as they are, so the plugin can modify their arguments in-place. 

889 Arguments that are only understood by the plugin should be popped 

890 or otherwise removed so that they aren't interpreted as erroneous 

891 arguments afterwards. 

892 

893 When the engine creation process completes and produces the 

894 :class:`.Engine` object, it is again passed to the plugin via the 

895 :meth:`.CreateEnginePlugin.engine_created` hook. In this hook, additional 

896 changes can be made to the engine, most typically involving setup of 

897 events (e.g. those defined in :ref:`core_event_toplevel`). 

898 

899 .. versionadded:: 1.1 

900 

901 """
