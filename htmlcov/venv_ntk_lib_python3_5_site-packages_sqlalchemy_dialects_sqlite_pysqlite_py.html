8r""" 

9.. dialect:: sqlite+pysqlite 

10 :name: pysqlite 

11 :dbapi: sqlite3 

12 :connectstring: sqlite+pysqlite:///file_path 

13 :url: http://docs.python.org/library/sqlite3.html 

14 

15 Note that ``pysqlite`` is the same driver as the ``sqlite3`` 

16 module included with the Python distribution. 

17 

18Driver 

19------ 

20 

21The ``sqlite3`` Python DBAPI is standard on all modern Python versions; 

22for cPython and Pypy, no additional installation is necessary. 

23 

24 

25Connect Strings 

26--------------- 

27 

28The file specification for the SQLite database is taken as the "database" 

29portion of the URL. Note that the format of a SQLAlchemy url is:: 

30 

31 driver://user:pass@host/database 

32 

33This means that the actual filename to be used starts with the characters to 

34the **right** of the third slash. So connecting to a relative filepath 

35looks like:: 

36 

37 # relative path 

38 e = create_engine('sqlite:///path/to/database.db') 

39 

40An absolute path, which is denoted by starting with a slash, means you 

41need **four** slashes:: 

42 

43 # absolute path 

44 e = create_engine('sqlite:////path/to/database.db') 

45 

46To use a Windows path, regular drive specifications and backslashes can be 

47used. Double backslashes are probably needed:: 

48 

49 # absolute path on Windows 

50 e = create_engine('sqlite:///C:\\path\\to\\database.db') 

51 

52The sqlite ``:memory:`` identifier is the default if no filepath is 

53present. Specify ``sqlite://`` and nothing else:: 

54 

55 # in-memory database 

56 e = create_engine('sqlite://') 

57 

58.. _pysqlite_uri_connections: 

59 

60URI Connections 

61^^^^^^^^^^^^^^^ 

62 

63Modern versions of SQLite support an alternative system of connecting using a 

64`driver level URI <https://www.sqlite.org/uri.html>`_, which has the advantage 

65that additional driver-level arguments can be passed including options such as 

66"read only". The Python sqlite3 driver supports this mode under modern Python 

673 versions. The SQLAlchemy pysqlite driver supports this mode of use by 

68specifing "uri=true" in the URL query string. The SQLite-level "URI" is kept 

69as the "database" portion of the SQLAlchemy url (that is, following a slash):: 

70 

71 e = create_engine("sqlite:///file:path/to/database?mode=ro&uri=true") 

72 

73.. note:: The "uri=true" parameter must appear in the **query string** 

74 of the URL. It will not currently work as expected if it is only 

75 present in the :paramref:`.create_engine.connect_args` parameter dictionary. 

76 

77The logic reconciles the simultaneous presence of SQLAlchemy's query string and 

78SQLite's query string by separating out the parameters that belong to the 

79Python sqlite3 driver vs. those that belong to the SQLite URI. This is 

80achieved through the use of a fixed list of parameters known to be accepted by 

81the Python side of the driver. For example, to include a URL that indicates 

82the Python sqlite3 "timeout" and "check_same_thread" parameters, along with the 

83SQLite "mode" and "nolock" parameters, they can all be passed together on the 

84query string:: 

85 

86 e = create_engine( 

87 "sqlite:///file:path/to/database?" 

88 "check_same_thread=true&timeout=10&mode=ro&nolock=1&uri=true" 

89 ) 

90 

91Above, the pysqlite / sqlite3 DBAPI would be passed arguments as:: 

92 

93 sqlite3.connect( 

94 "file:path/to/database?mode=ro&nolock=1", 

95 check_same_thread=True, timeout=10, uri=True 

96 ) 

97 

98Regarding future parameters added to either the Python or native drivers. new 

99parameter names added to the SQLite URI scheme should be automatically 

100accommodated by this scheme. New parameter names added to the Python driver 

101side can be accommodated by specifying them in the 

102:paramref:`.create_engine.connect_args` dictionary, until dialect support is 

103added by SQLAlchemy. For the less likely case that the native SQLite driver 

104adds a new parameter name that overlaps with one of the existing, known Python 

105driver parameters (such as "timeout" perhaps), SQLAlchemy's dialect would 

106require adjustment for the URL scheme to continue to support this. 

107 

108As is always the case for all SQLAlchemy dialects, the entire "URL" process 

109can be bypassed in :func:`.create_engine` through the use of the 

110:paramref:`.create_engine.creator` parameter which allows for a custom callable 

111that creates a Python sqlite3 driver level connection directly. 

112 

113.. versionadded:: 1.3.9 

114 

115.. seealso:: 

116 

117 `Uniform Resource Identifiers <https://www.sqlite.org/uri.html>`_ - in 

118 the SQLite documentation

120Compatibility with sqlite3 "native" date and datetime types 

121----------------------------------------------------------- 

122 

123The pysqlite driver includes the sqlite3.PARSE_DECLTYPES and 

124sqlite3.PARSE_COLNAMES options, which have the effect of any column 

125or expression explicitly cast as "date" or "timestamp" will be converted 

126to a Python date or datetime object. The date and datetime types provided 

127with the pysqlite dialect are not currently compatible with these options, 

128since they render the ISO date/datetime including microseconds, which 

129pysqlite's driver does not. Additionally, SQLAlchemy does not at 

130this time automatically render the "cast" syntax required for the 

131freestanding functions "current_timestamp" and "current_date" to return 

132datetime/date types natively. Unfortunately, pysqlite 

133does not provide the standard DBAPI types in ``cursor.description``, 

134leaving SQLAlchemy with no way to detect these types on the fly 

135without expensive per-row type checks. 

136 

137Keeping in mind that pysqlite's parsing option is not recommended, 

138nor should be necessary, for use with SQLAlchemy, usage of PARSE_DECLTYPES 

139can be forced if one configures "native_datetime=True" on create_engine():: 

140 

141 engine = create_engine('sqlite://', 

142 connect_args={'detect_types': 

143 sqlite3.PARSE_DECLTYPES|sqlite3.PARSE_COLNAMES}, 

144 native_datetime=True 

145 ) 

146 

147With this flag enabled, the DATE and TIMESTAMP types (but note - not the 

148DATETIME or TIME types...confused yet ?) will not perform any bind parameter 

149or result processing. Execution of "func.current_date()" will return a string. 

150"func.current_timestamp()" is registered as returning a DATETIME type in 

151SQLAlchemy, so this function still receives SQLAlchemy-level result 

152processing.

154.. _pysqlite_threading_pooling: 

155 

156Threading/Pooling Behavior 

157--------------------------- 

158 

159Pysqlite's default behavior is to prohibit the usage of a single connection 

160in more than one thread. This is originally intended to work with older 

161versions of SQLite that did not support multithreaded operation under 

162various circumstances. In particular, older SQLite versions 

163did not allow a ``:memory:`` database to be used in multiple threads 

164under any circumstances. 

165 

166Pysqlite does include a now-undocumented flag known as 

167``check_same_thread`` which will disable this check, however note that 

168pysqlite connections are still not safe to use in concurrently in multiple 

169threads. In particular, any statement execution calls would need to be 

170externally mutexed, as Pysqlite does not provide for thread-safe propagation 

171of error messages among other things. So while even ``:memory:`` databases 

172can be shared among threads in modern SQLite, Pysqlite doesn't provide enough 

173thread-safety to make this usage worth it. 

174 

175SQLAlchemy sets up pooling to work with Pysqlite's default behavior: 

176 

177* When a ``:memory:`` SQLite database is specified, the dialect by default 

178 will use :class:`.SingletonThreadPool`. This pool maintains a single 

179 connection per thread, so that all access to the engine within the current 

180 thread use the same ``:memory:`` database - other threads would access a 

181 different ``:memory:`` database. 

182* When a file-based database is specified, the dialect will use 

183 :class:`.NullPool` as the source of connections. This pool closes and 

184 discards connections which are returned to the pool immediately. SQLite 

185 file-based connections have extremely low overhead, so pooling is not 

186 necessary. The scheme also prevents a connection from being used again in 

187 a different thread and works best with SQLite's coarse-grained file locking. 

188 

189Using a Memory Database in Multiple Threads 

190^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ 

191 

192To use a ``:memory:`` database in a multithreaded scenario, the same 

193connection object must be shared among threads, since the database exists 

194only within the scope of that connection. The 

195:class:`.StaticPool` implementation will maintain a single connection 

196globally, and the ``check_same_thread`` flag can be passed to Pysqlite 

197as ``False``:: 

198 

199 from sqlalchemy.pool import StaticPool 

200 engine = create_engine('sqlite://', 

201 connect_args={'check_same_thread':False}, 

202 poolclass=StaticPool) 

203 

204Note that using a ``:memory:`` database in multiple threads requires a recent 

205version of SQLite. 

206 

207Using Temporary Tables with SQLite 

208^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ 

209 

210Due to the way SQLite deals with temporary tables, if you wish to use a 

211temporary table in a file-based SQLite database across multiple checkouts 

212from the connection pool, such as when using an ORM :class:`.Session` where 

213the temporary table should continue to remain after :meth:`.Session.commit` or 

214:meth:`.Session.rollback` is called, a pool which maintains a single 

215connection must be used. Use :class:`.SingletonThreadPool` if the scope is 

216only needed within the current thread, or :class:`.StaticPool` is scope is 

217needed within multiple threads for this case:: 

218 

219 # maintain the same connection per thread 

220 from sqlalchemy.pool import SingletonThreadPool 

221 engine = create_engine('sqlite:///mydb.db', 

222 poolclass=SingletonThreadPool) 

223 

224 

225 # maintain the same connection across all threads 

226 from sqlalchemy.pool import StaticPool 

227 engine = create_engine('sqlite:///mydb.db', 

228 poolclass=StaticPool) 

229 

230Note that :class:`.SingletonThreadPool` should be configured for the number 

231of threads that are to be used; beyond that number, connections will be 

232closed out in a non deterministic way. 

233 

234Unicode 

235------- 

236 

237The pysqlite driver only returns Python ``unicode`` objects in result sets, 

238never plain strings, and accommodates ``unicode`` objects within bound 

239parameter values in all cases. Regardless of the SQLAlchemy string type in 

240use, string-based result values will by Python ``unicode`` in Python 2. 

241The :class:`.Unicode` type should still be used to indicate those columns that 

242require unicode, however, so that non-``unicode`` values passed inadvertently 

243will emit a warning. Pysqlite will emit an error if a non-``unicode`` string 

244is passed containing non-ASCII characters. 

245 

246.. _pysqlite_serializable: 

247 

248Serializable isolation / Savepoints / Transactional DDL 

249------------------------------------------------------- 

250 

251In the section :ref:`sqlite_concurrency`, we refer to the pysqlite 

252driver's assortment of issues that prevent several features of SQLite 

253from working correctly. The pysqlite DBAPI driver has several 

254long-standing bugs which impact the correctness of its transactional 

255behavior. In its default mode of operation, SQLite features such as 

256SERIALIZABLE isolation, transactional DDL, and SAVEPOINT support are 

257non-functional, and in order to use these features, workarounds must 

258be taken. 

259 

260The issue is essentially that the driver attempts to second-guess the user's 

261intent, failing to start transactions and sometimes ending them prematurely, in 

262an effort to minimize the SQLite databases's file locking behavior, even 

263though SQLite itself uses "shared" locks for read-only activities. 

264 

265SQLAlchemy chooses to not alter this behavior by default, as it is the 

266long-expected behavior of the pysqlite driver; if and when the pysqlite 

267driver attempts to repair these issues, that will be more of a driver towards 

268defaults for SQLAlchemy. 

269 

270The good news is that with a few events, we can implement transactional 

271support fully, by disabling pysqlite's feature entirely and emitting BEGIN 

272ourselves. This is achieved using two event listeners:: 

273 

274 from sqlalchemy import create_engine, event 

275 

276 engine = create_engine("sqlite:///myfile.db") 

277 

278 @event.listens_for(engine, "connect") 

279 def do_connect(dbapi_connection, connection_record): 

280 # disable pysqlite's emitting of the BEGIN statement entirely. 

281 # also stops it from emitting COMMIT before any DDL. 

282 dbapi_connection.isolation_level = None 

283 

284 @event.listens_for(engine, "begin") 

285 def do_begin(conn): 

286 # emit our own BEGIN 

287 conn.execute("BEGIN") 

288 

289Above, we intercept a new pysqlite connection and disable any transactional 

290integration. Then, at the point at which SQLAlchemy knows that transaction 

291scope is to begin, we emit ``"BEGIN"`` ourselves. 

292 

293When we take control of ``"BEGIN"``, we can also control directly SQLite's 

294locking modes, introduced at 

295`BEGIN TRANSACTION <http://sqlite.org/lang_transaction.html>`_, 

296by adding the desired locking mode to our ``"BEGIN"``:: 

297 

298 @event.listens_for(engine, "begin") 

299 def do_begin(conn): 

300 conn.execute("BEGIN EXCLUSIVE") 

301 

302.. seealso:: 

303 

304 `BEGIN TRANSACTION <http://sqlite.org/lang_transaction.html>`_ - 

305 on the SQLite site 

306 

307 `sqlite3 SELECT does not BEGIN a transaction <http://bugs.python.org/issue9924>`_ - 

308 on the Python bug tracker 

309 

310 `sqlite3 module breaks transactions and potentially corrupts data <http://bugs.python.org/issue10740>`_ - 

311 on the Python bug tracker
