14Sequences/SERIAL/IDENTITY 

15------------------------- 

16 

17PostgreSQL supports sequences, and SQLAlchemy uses these as the default means 

18of creating new primary key values for integer-based primary key columns. When 

19creating tables, SQLAlchemy will issue the ``SERIAL`` datatype for 

20integer-based primary key columns, which generates a sequence and server side 

21default corresponding to the column. 

22 

23To specify a specific named sequence to be used for primary key generation, 

24use the :func:`~sqlalchemy.schema.Sequence` construct:: 

25 

26 Table('sometable', metadata, 

27 Column('id', Integer, Sequence('some_id_seq'), primary_key=True) 

28 ) 

29 

30When SQLAlchemy issues a single INSERT statement, to fulfill the contract of 

31having the "last insert identifier" available, a RETURNING clause is added to 

32the INSERT statement which specifies the primary key columns should be 

33returned after the statement completes. The RETURNING functionality only takes 

34place if PostgreSQL 8.2 or later is in use. As a fallback approach, the 

35sequence, whether specified explicitly or implicitly via ``SERIAL``, is 

36executed independently beforehand, the returned value to be used in the 

37subsequent insert. Note that when an 

38:func:`~sqlalchemy.sql.expression.insert()` construct is executed using 

39"executemany" semantics, the "last inserted identifier" functionality does not 

40apply; no RETURNING clause is emitted nor is the sequence pre-executed in this 

41case. 

42 

43To force the usage of RETURNING by default off, specify the flag 

44``implicit_returning=False`` to :func:`.create_engine`. 

45 

46PostgreSQL 10 IDENTITY columns 

47^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ 

48 

49PostgreSQL 10 has a new IDENTITY feature that supersedes the use of SERIAL. 

50Built-in support for rendering of IDENTITY is not available yet, however the 

51following compilation hook may be used to replace occurrences of SERIAL with 

52IDENTITY:: 

53 

54 from sqlalchemy.schema import CreateColumn 

55 from sqlalchemy.ext.compiler import compiles 

56 

57 

58 @compiles(CreateColumn, 'postgresql') 

59 def use_identity(element, compiler, **kw): 

60 text = compiler.visit_create_column(element, **kw) 

61 text = text.replace("SERIAL", "INT GENERATED BY DEFAULT AS IDENTITY") 

62 return text 

63 

64Using the above, a table such as:: 

65 

66 t = Table( 

67 't', m, 

68 Column('id', Integer, primary_key=True), 

69 Column('data', String) 

70 ) 

71 

72Will generate on the backing database as:: 

73 

74 CREATE TABLE t ( 

75 id INT GENERATED BY DEFAULT AS IDENTITY NOT NULL, 

76 data VARCHAR, 

77 PRIMARY KEY (id) 

78 )

82Transaction Isolation Level 

83--------------------------- 

84 

85All PostgreSQL dialects support setting of transaction isolation level 

86both via a dialect-specific parameter 

87:paramref:`.create_engine.isolation_level` accepted by :func:`.create_engine`, 

88as well as the :paramref:`.Connection.execution_options.isolation_level` 

89argument as passed to :meth:`.Connection.execution_options`. 

90When using a non-psycopg2 dialect, this feature works by issuing the command 

91``SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL <level>`` for 

92each new connection. For the special AUTOCOMMIT isolation level, 

93DBAPI-specific techniques are used. 

94 

95To set isolation level using :func:`.create_engine`:: 

96 

97 engine = create_engine( 

98 "postgresql+pg8000://scott:tiger@localhost/test", 

99 isolation_level="READ UNCOMMITTED" 

100 ) 

101 

102To set using per-connection execution options:: 

103 

104 connection = engine.connect() 

105 connection = connection.execution_options( 

106 isolation_level="READ COMMITTED" 

107 ) 

108 

109Valid values for ``isolation_level`` include: 

110 

111* ``READ COMMITTED`` 

112* ``READ UNCOMMITTED`` 

113* ``REPEATABLE READ`` 

114* ``SERIALIZABLE`` 

115* ``AUTOCOMMIT`` - on psycopg2 / pg8000 only

125Remote-Schema Table Introspection and PostgreSQL search_path 

126------------------------------------------------------------ 

127 

128**TL;DR;**: keep the ``search_path`` variable set to its default of ``public``, 

129name schemas **other** than ``public`` explicitly within ``Table`` definitions. 

130 

131The PostgreSQL dialect can reflect tables from any schema. The 

132:paramref:`.Table.schema` argument, or alternatively the 

133:paramref:`.MetaData.reflect.schema` argument determines which schema will 

134be searched for the table or tables. The reflected :class:`.Table` objects 

135will in all cases retain this ``.schema`` attribute as was specified. 

136However, with regards to tables which these :class:`.Table` objects refer to 

137via foreign key constraint, a decision must be made as to how the ``.schema`` 

138is represented in those remote tables, in the case where that remote 

139schema name is also a member of the current 

140`PostgreSQL search path 

141<http://www.postgresql.org/docs/current/static/ddl-schemas.html#DDL-SCHEMAS-PATH>`_. 

142 

143By default, the PostgreSQL dialect mimics the behavior encouraged by 

144PostgreSQL's own ``pg_get_constraintdef()`` builtin procedure. This function 

145returns a sample definition for a particular foreign key constraint, 

146omitting the referenced schema name from that definition when the name is 

147also in the PostgreSQL schema search path. The interaction below 

148illustrates this behavior:: 

149 

150 test=> CREATE TABLE test_schema.referred(id INTEGER PRIMARY KEY); 

151 CREATE TABLE 

152 test=> CREATE TABLE referring( 

153 test(> id INTEGER PRIMARY KEY, 

154 test(> referred_id INTEGER REFERENCES test_schema.referred(id)); 

155 CREATE TABLE 

156 test=> SET search_path TO public, test_schema; 

157 test=> SELECT pg_catalog.pg_get_constraintdef(r.oid, true) FROM 

158 test-> pg_catalog.pg_class c JOIN pg_catalog.pg_namespace n 

159 test-> ON n.oid = c.relnamespace 

160 test-> JOIN pg_catalog.pg_constraint r ON c.oid = r.conrelid 

161 test-> WHERE c.relname='referring' AND r.contype = 'f' 

162 test-> ; 

163 pg_get_constraintdef 

164 --------------------------------------------------- 

165 FOREIGN KEY (referred_id) REFERENCES referred(id) 

166 (1 row) 

167 

168Above, we created a table ``referred`` as a member of the remote schema 

169``test_schema``, however when we added ``test_schema`` to the 

170PG ``search_path`` and then asked ``pg_get_constraintdef()`` for the 

171``FOREIGN KEY`` syntax, ``test_schema`` was not included in the output of 

172the function. 

173 

174On the other hand, if we set the search path back to the typical default 

175of ``public``:: 

176 

177 test=> SET search_path TO public; 

178 SET 

179 

180The same query against ``pg_get_constraintdef()`` now returns the fully 

181schema-qualified name for us:: 

182 

183 test=> SELECT pg_catalog.pg_get_constraintdef(r.oid, true) FROM 

184 test-> pg_catalog.pg_class c JOIN pg_catalog.pg_namespace n 

185 test-> ON n.oid = c.relnamespace 

186 test-> JOIN pg_catalog.pg_constraint r ON c.oid = r.conrelid 

187 test-> WHERE c.relname='referring' AND r.contype = 'f'; 

188 pg_get_constraintdef 

189 --------------------------------------------------------------- 

190 FOREIGN KEY (referred_id) REFERENCES test_schema.referred(id) 

191 (1 row) 

192 

193SQLAlchemy will by default use the return value of ``pg_get_constraintdef()`` 

194in order to determine the remote schema name. That is, if our ``search_path`` 

195were set to include ``test_schema``, and we invoked a table 

196reflection process as follows:: 

197 

198 >>> from sqlalchemy import Table, MetaData, create_engine 

199 >>> engine = create_engine("postgresql://scott:tiger@localhost/test") 

200 >>> with engine.connect() as conn: 

201 ... conn.execute("SET search_path TO test_schema, public") 

202 ... meta = MetaData() 

203 ... referring = Table('referring', meta, 

204 ... autoload=True, autoload_with=conn) 

205 ... 

206 <sqlalchemy.engine.result.ResultProxy object at 0x101612ed0> 

207 

208The above process would deliver to the :attr:`.MetaData.tables` collection 

209``referred`` table named **without** the schema:: 

210 

211 >>> meta.tables['referred'].schema is None 

212 True 

213 

214To alter the behavior of reflection such that the referred schema is 

215maintained regardless of the ``search_path`` setting, use the 

216``postgresql_ignore_search_path`` option, which can be specified as a 

217dialect-specific argument to both :class:`.Table` as well as 

218:meth:`.MetaData.reflect`:: 

219 

220 >>> with engine.connect() as conn: 

221 ... conn.execute("SET search_path TO test_schema, public") 

222 ... meta = MetaData() 

223 ... referring = Table('referring', meta, autoload=True, 

224 ... autoload_with=conn, 

225 ... postgresql_ignore_search_path=True) 

226 ... 

227 <sqlalchemy.engine.result.ResultProxy object at 0x1016126d0> 

228 

229We will now have ``test_schema.referred`` stored as schema-qualified:: 

230 

231 >>> meta.tables['test_schema.referred'].schema 

232 'test_schema' 

233 

234.. sidebar:: Best Practices for PostgreSQL Schema reflection 

235 

236 The description of PostgreSQL schema reflection behavior is complex, and 

237 is the product of many years of dealing with widely varied use cases and 

238 user preferences. But in fact, there's no need to understand any of it if 

239 you just stick to the simplest use pattern: leave the ``search_path`` set 

240 to its default of ``public`` only, never refer to the name ``public`` as 

241 an explicit schema name otherwise, and refer to all other schema names 

242 explicitly when building up a :class:`.Table` object. The options 

243 described here are only for those users who can't, or prefer not to, stay 

244 within these guidelines. 

245 

246Note that **in all cases**, the "default" schema is always reflected as 

247``None``. The "default" schema on PostgreSQL is that which is returned by the 

248PostgreSQL ``current_schema()`` function. On a typical PostgreSQL 

249installation, this is the name ``public``. So a table that refers to another 

250which is in the ``public`` (i.e. default) schema will always have the 

251``.schema`` attribute set to ``None``.

264INSERT/UPDATE...RETURNING 

265------------------------- 

266 

267The dialect supports PG 8.2's ``INSERT..RETURNING``, ``UPDATE..RETURNING`` and 

268``DELETE..RETURNING`` syntaxes. ``INSERT..RETURNING`` is used by default 

269for single-row INSERT statements in order to fetch newly generated 

270primary key identifiers. To specify an explicit ``RETURNING`` clause, 

271use the :meth:`._UpdateBase.returning` method on a per-statement basis:: 

272 

273 # INSERT..RETURNING 

274 result = table.insert().returning(table.c.col1, table.c.col2).\ 

275 values(name='foo') 

276 print result.fetchall() 

277 

278 # UPDATE..RETURNING 

279 result = table.update().returning(table.c.col1, table.c.col2).\ 

280 where(table.c.name=='foo').values(name='bar') 

281 print result.fetchall() 

282 

283 # DELETE..RETURNING 

284 result = table.delete().returning(table.c.col1, table.c.col2).\ 

285 where(table.c.name=='foo') 

286 print result.fetchall()

290INSERT...ON CONFLICT (Upsert) 

291------------------------------ 

292 

293Starting with version 9.5, PostgreSQL allows "upserts" (update or insert) of 

294rows into a table via the ``ON CONFLICT`` clause of the ``INSERT`` statement. A 

295candidate row will only be inserted if that row does not violate any unique 

296constraints. In the case of a unique constraint violation, a secondary action 

297can occur which can be either "DO UPDATE", indicating that the data in the 

298target row should be updated, or "DO NOTHING", which indicates to silently skip 

299this row. 

300 

301Conflicts are determined using existing unique constraints and indexes. These 

302constraints may be identified either using their name as stated in DDL, 

303or they may be *inferred* by stating the columns and conditions that comprise 

304the indexes. 

305 

306SQLAlchemy provides ``ON CONFLICT`` support via the PostgreSQL-specific 

307:func:`.postgresql.dml.insert()` function, which provides 

308the generative methods :meth:`~.postgresql.dml.Insert.on_conflict_do_update` 

309and :meth:`~.postgresql.dml.Insert.on_conflict_do_nothing`:: 

310 

311 from sqlalchemy.dialects.postgresql import insert 

312 

313 insert_stmt = insert(my_table).values( 

314 id='some_existing_id', 

315 data='inserted value') 

316 

317 do_nothing_stmt = insert_stmt.on_conflict_do_nothing( 

318 index_elements=['id'] 

319 ) 

320 

321 conn.execute(do_nothing_stmt) 

322 

323 do_update_stmt = insert_stmt.on_conflict_do_update( 

324 constraint='pk_my_table', 

325 set_=dict(data='updated value') 

326 ) 

327 

328 conn.execute(do_update_stmt) 

329 

330Both methods supply the "target" of the conflict using either the 

331named constraint or by column inference: 

332 

333* The :paramref:`.Insert.on_conflict_do_update.index_elements` argument 

334 specifies a sequence containing string column names, :class:`.Column` 

335 objects, and/or SQL expression elements, which would identify a unique 

336 index:: 

337 

338 do_update_stmt = insert_stmt.on_conflict_do_update( 

339 index_elements=['id'], 

340 set_=dict(data='updated value') 

341 ) 

342 

343 do_update_stmt = insert_stmt.on_conflict_do_update( 

344 index_elements=[my_table.c.id], 

345 set_=dict(data='updated value') 

346 ) 

347 

348* When using :paramref:`.Insert.on_conflict_do_update.index_elements` to 

349 infer an index, a partial index can be inferred by also specifying the 

350 use the :paramref:`.Insert.on_conflict_do_update.index_where` parameter:: 

351 

352 from sqlalchemy.dialects.postgresql import insert 

353 

354 stmt = insert(my_table).values(user_email='a@b.com', data='inserted data') 

355 stmt = stmt.on_conflict_do_update( 

356 index_elements=[my_table.c.user_email], 

357 index_where=my_table.c.user_email.like('%@gmail.com'), 

358 set_=dict(data=stmt.excluded.data) 

359 ) 

360 conn.execute(stmt) 

361 

362* The :paramref:`.Insert.on_conflict_do_update.constraint` argument is 

363 used to specify an index directly rather than inferring it. This can be 

364 the name of a UNIQUE constraint, a PRIMARY KEY constraint, or an INDEX:: 

365 

366 do_update_stmt = insert_stmt.on_conflict_do_update( 

367 constraint='my_table_idx_1', 

368 set_=dict(data='updated value') 

369 ) 

370 

371 do_update_stmt = insert_stmt.on_conflict_do_update( 

372 constraint='my_table_pk', 

373 set_=dict(data='updated value') 

374 ) 

375 

376* The :paramref:`.Insert.on_conflict_do_update.constraint` argument may 

377 also refer to a SQLAlchemy construct representing a constraint, 

378 e.g. :class:`.UniqueConstraint`, :class:`.PrimaryKeyConstraint`, 

379 :class:`.Index`, or :class:`.ExcludeConstraint`. In this use, 

380 if the constraint has a name, it is used directly. Otherwise, if the 

381 constraint is unnamed, then inference will be used, where the expressions 

382 and optional WHERE clause of the constraint will be spelled out in the 

383 construct. This use is especially convenient 

384 to refer to the named or unnamed primary key of a :class:`.Table` using the 

385 :attr:`.Table.primary_key` attribute:: 

386 

387 do_update_stmt = insert_stmt.on_conflict_do_update( 

388 constraint=my_table.primary_key, 

389 set_=dict(data='updated value') 

390 ) 

391 

392``ON CONFLICT...DO UPDATE`` is used to perform an update of the already 

393existing row, using any combination of new values as well as values 

394from the proposed insertion. These values are specified using the 

395:paramref:`.Insert.on_conflict_do_update.set_` parameter. This 

396parameter accepts a dictionary which consists of direct values 

397for UPDATE:: 

398 

399 from sqlalchemy.dialects.postgresql import insert 

400 

401 stmt = insert(my_table).values(id='some_id', data='inserted value') 

402 do_update_stmt = stmt.on_conflict_do_update( 

403 index_elements=['id'], 

404 set_=dict(data='updated value') 

405 ) 

406 conn.execute(do_update_stmt) 

407 

408.. warning:: 

409 

410 The :meth:`.Insert.on_conflict_do_update` method does **not** take into 

411 account Python-side default UPDATE values or generation functions, e.g. 

412 those specified using :paramref:`.Column.onupdate`. 

413 These values will not be exercised for an ON CONFLICT style of UPDATE, 

414 unless they are manually specified in the 

415 :paramref:`.Insert.on_conflict_do_update.set_` dictionary. 

416 

417In order to refer to the proposed insertion row, the special alias 

418:attr:`~.postgresql.dml.Insert.excluded` is available as an attribute on 

419the :class:`.postgresql.dml.Insert` object; this object is a 

420:class:`.ColumnCollection` which alias contains all columns of the target 

421table:: 

422 

423 from sqlalchemy.dialects.postgresql import insert 

424 

425 stmt = insert(my_table).values( 

426 id='some_id', 

427 data='inserted value', 

428 author='jlh') 

429 do_update_stmt = stmt.on_conflict_do_update( 

430 index_elements=['id'], 

431 set_=dict(data='updated value', author=stmt.excluded.author) 

432 ) 

433 conn.execute(do_update_stmt) 

434 

435The :meth:`.Insert.on_conflict_do_update` method also accepts 

436a WHERE clause using the :paramref:`.Insert.on_conflict_do_update.where` 

437parameter, which will limit those rows which receive an UPDATE:: 

438 

439 from sqlalchemy.dialects.postgresql import insert 

440 

441 stmt = insert(my_table).values( 

442 id='some_id', 

443 data='inserted value', 

444 author='jlh') 

445 on_update_stmt = stmt.on_conflict_do_update( 

446 index_elements=['id'], 

447 set_=dict(data='updated value', author=stmt.excluded.author) 

448 where=(my_table.c.status == 2) 

449 ) 

450 conn.execute(on_update_stmt) 

451 

452``ON CONFLICT`` may also be used to skip inserting a row entirely 

453if any conflict with a unique or exclusion constraint occurs; below 

454this is illustrated using the 

455:meth:`~.postgresql.dml.Insert.on_conflict_do_nothing` method:: 

456 

457 from sqlalchemy.dialects.postgresql import insert 

458 

459 stmt = insert(my_table).values(id='some_id', data='inserted value') 

460 stmt = stmt.on_conflict_do_nothing(index_elements=['id']) 

461 conn.execute(stmt) 

462 

463If ``DO NOTHING`` is used without specifying any columns or constraint, 

464it has the effect of skipping the INSERT for any unique or exclusion 

465constraint violation which occurs:: 

466 

467 from sqlalchemy.dialects.postgresql import insert 

468 

469 stmt = insert(my_table).values(id='some_id', data='inserted value') 

470 stmt = stmt.on_conflict_do_nothing() 

471 conn.execute(stmt)

483Full Text Search 

484---------------- 

485 

486SQLAlchemy makes available the PostgreSQL ``@@`` operator via the 

487:meth:`.ColumnElement.match` method on any textual column expression. 

488On a PostgreSQL dialect, an expression like the following:: 

489 

490 select([sometable.c.text.match("search string")]) 

491 

492will emit to the database:: 

493 

494 SELECT text @@ to_tsquery('search string') FROM table 

495 

496The PostgreSQL text search functions such as ``to_tsquery()`` 

497and ``to_tsvector()`` are available 

498explicitly using the standard :data:`.func` construct. For example:: 

499 

500 select([ 

501 func.to_tsvector('fat cats ate rats').match('cat & rat') 

502 ]) 

503 

504Emits the equivalent of:: 

505 

506 SELECT to_tsvector('fat cats ate rats') @@ to_tsquery('cat & rat') 

507 

508The :class:`.postgresql.TSVECTOR` type can provide for explicit CAST:: 

509 

510 from sqlalchemy.dialects.postgresql import TSVECTOR 

511 from sqlalchemy import select, cast 

512 select([cast("some text", TSVECTOR)]) 

513 

514produces a statement equivalent to:: 

515 

516 SELECT CAST('some text' AS TSVECTOR) AS anon_1 

517 

518Full Text Searches in PostgreSQL are influenced by a combination of: the 

519PostgreSQL setting of ``default_text_search_config``, the ``regconfig`` used 

520to build the GIN/GiST indexes, and the ``regconfig`` optionally passed in 

521during a query. 

522 

523When performing a Full Text Search against a column that has a GIN or 

524GiST index that is already pre-computed (which is common on full text 

525searches) one may need to explicitly pass in a particular PostgreSQL 

526``regconfig`` value to ensure the query-planner utilizes the index and does 

527not re-compute the column on demand. 

528 

529In order to provide for this explicit query planning, or to use different 

530search strategies, the ``match`` method accepts a ``postgresql_regconfig`` 

531keyword argument:: 

532 

533 select([mytable.c.id]).where( 

534 mytable.c.title.match('somestring', postgresql_regconfig='english') 

535 ) 

536 

537Emits the equivalent of:: 

538 

539 SELECT mytable.id FROM mytable 

540 WHERE mytable.title @@ to_tsquery('english', 'somestring') 

541 

542One can also specifically pass in a `'regconfig'` value to the 

543``to_tsvector()`` command as the initial argument:: 

544 

545 select([mytable.c.id]).where( 

546 func.to_tsvector('english', mytable.c.title )\ 

547 .match('somestring', postgresql_regconfig='english') 

548 ) 

549 

550produces a statement equivalent to:: 

551 

552 SELECT mytable.id FROM mytable 

553 WHERE to_tsvector('english', mytable.title) @@ 

554 to_tsquery('english', 'somestring') 

555 

556It is recommended that you use the ``EXPLAIN ANALYZE...`` tool from 

557PostgreSQL to ensure that you are generating queries with SQLAlchemy that 

558take full advantage of any indexes you may have created for full text search.

560FROM ONLY ... 

561------------------------ 

562 

563The dialect supports PostgreSQL's ONLY keyword for targeting only a particular 

564table in an inheritance hierarchy. This can be used to produce the 

565``SELECT ... FROM ONLY``, ``UPDATE ONLY ...``, and ``DELETE FROM ONLY ...`` 

566syntaxes. It uses SQLAlchemy's hints mechanism:: 

567 

568 # SELECT ... FROM ONLY ... 

569 result = table.select().with_hint(table, 'ONLY', 'postgresql') 

570 print result.fetchall() 

571 

572 # UPDATE ONLY ... 

573 table.update(values=dict(foo='bar')).with_hint('ONLY', 

574 dialect_name='postgresql') 

575 

576 # DELETE FROM ONLY ... 

577 table.delete().with_hint('ONLY', dialect_name='postgresql')

590Partial Indexes 

591^^^^^^^^^^^^^^^ 

592 

593Partial indexes add criterion to the index definition so that the index is 

594applied to a subset of rows. These can be specified on :class:`.Index` 

595using the ``postgresql_where`` keyword argument:: 

596 

597 Index('my_index', my_table.c.id, postgresql_where=my_table.c.value > 10)

599Operator Classes 

600^^^^^^^^^^^^^^^^ 

601 

602PostgreSQL allows the specification of an *operator class* for each column of 

603an index (see 

604http://www.postgresql.org/docs/8.3/interactive/indexes-opclass.html). 

605The :class:`.Index` construct allows these to be specified via the 

606``postgresql_ops`` keyword argument:: 

607 

608 Index( 

609 'my_index', my_table.c.id, my_table.c.data, 

610 postgresql_ops={ 

611 'data': 'text_pattern_ops', 

612 'id': 'int4_ops' 

613 }) 

614 

615Note that the keys in the ``postgresql_ops`` dictionary are the "key" name of 

616the :class:`.Column`, i.e. the name used to access it from the ``.c`` 

617collection of :class:`.Table`, which can be configured to be different than 

618the actual name of the column as expressed in the database. 

619 

620If ``postgresql_ops`` is to be used against a complex SQL expression such 

621as a function call, then to apply to the column it must be given a label 

622that is identified in the dictionary by name, e.g.:: 

623 

624 Index( 

625 'my_index', my_table.c.id, 

626 func.lower(my_table.c.data).label('data_lower'), 

627 postgresql_ops={ 

628 'data_lower': 'text_pattern_ops', 

629 'id': 'int4_ops' 

630 })

633Index Types 

634^^^^^^^^^^^ 

635 

636PostgreSQL provides several index types: B-Tree, Hash, GiST, and GIN, as well 

637as the ability for users to create their own (see 

638http://www.postgresql.org/docs/8.3/static/indexes-types.html). These can be 

639specified on :class:`.Index` using the ``postgresql_using`` keyword argument:: 

640 

641 Index('my_index', my_table.c.data, postgresql_using='gin') 

642 

643The value passed to the keyword argument will be simply passed through to the 

644underlying CREATE INDEX command, so it *must* be a valid index type for your 

645version of PostgreSQL.

649Index Storage Parameters 

650^^^^^^^^^^^^^^^^^^^^^^^^ 

651 

652PostgreSQL allows storage parameters to be set on indexes. The storage 

653parameters available depend on the index method used by the index. Storage 

654parameters can be specified on :class:`.Index` using the ``postgresql_with`` 

655keyword argument:: 

656 

657 Index('my_index', my_table.c.data, postgresql_with={"fillfactor": 50}) 

658 

659.. versionadded:: 1.0.6 

660 

661PostgreSQL allows to define the tablespace in which to create the index. 

662The tablespace can be specified on :class:`.Index` using the 

663``postgresql_tablespace`` keyword argument:: 

664 

665 Index('my_index', my_table.c.data, postgresql_tablespace='my_tablespace') 

666 

667.. versionadded:: 1.1 

668 

669Note that the same option is available on :class:`.Table` as well.

671.. _postgresql_index_concurrently: 

672 

673Indexes with CONCURRENTLY 

674^^^^^^^^^^^^^^^^^^^^^^^^^ 

675 

676The PostgreSQL index option CONCURRENTLY is supported by passing the 

677flag ``postgresql_concurrently`` to the :class:`.Index` construct:: 

678 

679 tbl = Table('testtbl', m, Column('data', Integer)) 

680 

681 idx1 = Index('test_idx1', tbl.c.data, postgresql_concurrently=True) 

682 

683The above index construct will render DDL for CREATE INDEX, assuming 

684PostgreSQL 8.2 or higher is detected or for a connection-less dialect, as:: 

685 

686 CREATE INDEX CONCURRENTLY test_idx1 ON testtbl (data) 

687 

688For DROP INDEX, assuming PostgreSQL 9.2 or higher is detected or for 

689a connection-less dialect, it will emit:: 

690 

691 DROP INDEX CONCURRENTLY test_idx1 

692 

693.. versionadded:: 1.1 support for CONCURRENTLY on DROP INDEX. The 

694 CONCURRENTLY keyword is now only emitted if a high enough version 

695 of PostgreSQL is detected on the connection (or for a connection-less 

696 dialect). 

697 

698When using CONCURRENTLY, the PostgreSQL database requires that the statement 

699be invoked outside of a transaction block. The Python DBAPI enforces that 

700even for a single statement, a transaction is present, so to use this 

701construct, the DBAPI's "autocommit" mode must be used:: 

702 

703 metadata = MetaData() 

704 table = Table( 

705 "foo", metadata, 

706 Column("id", String)) 

707 index = Index( 

708 "foo_idx", table.c.id, postgresql_concurrently=True) 

709 

710 with engine.connect() as conn: 

711 with conn.execution_options(isolation_level='AUTOCOMMIT'): 

712 table.create(conn) 

713

715 

716 :ref:`postgresql_isolation_level`

720PostgreSQL Index Reflection 

721--------------------------- 

722 

723The PostgreSQL database creates a UNIQUE INDEX implicitly whenever the 

724UNIQUE CONSTRAINT construct is used. When inspecting a table using 

725:class:`.Inspector`, the :meth:`.Inspector.get_indexes` 

726and the :meth:`.Inspector.get_unique_constraints` will report on these 

727two constructs distinctly; in the case of the index, the key 

728``duplicates_constraint`` will be present in the index entry if it is 

729detected as mirroring a constraint. When performing reflection using 

730``Table(..., autoload=True)``, the UNIQUE INDEX is **not** returned 

731in :attr:`.Table.indexes` when it is detected as mirroring a 

732:class:`.UniqueConstraint` in the :attr:`.Table.constraints` collection. 

733 

734.. versionchanged:: 1.0.0 - :class:`.Table` reflection now includes 

735 :class:`.UniqueConstraint` objects present in the :attr:`.Table.constraints` 

736 collection; the PostgreSQL backend will no longer include a "mirrored" 

737 :class:`.Index` construct in :attr:`.Table.indexes` if it is detected 

738 as corresponding to a unique constraint.

855Using ENUM with ARRAY 

856^^^^^^^^^^^^^^^^^^^^^ 

857 

858The combination of ENUM and ARRAY is not directly supported by backend 

859DBAPIs at this time. In order to send and receive an ARRAY of ENUM, 

860use the following workaround type, which decorates the 

861:class:`.postgresql.ARRAY` datatype. 

862 

863.. sourcecode:: python 

864 

865 from sqlalchemy import TypeDecorator 

866 from sqlalchemy.dialects.postgresql import ARRAY 

867 

868 class ArrayOfEnum(TypeDecorator): 

869 impl = ARRAY 

870 

871 def bind_expression(self, bindvalue): 

872 return sa.cast(bindvalue, self) 

873 

874 def result_processor(self, dialect, coltype): 

875 super_rp = super(ArrayOfEnum, self).result_processor( 

876 dialect, coltype) 

877 

878 def handle_raw_string(value): 

879 inner = re.match(r"^{(.*)}$", value).group(1) 

880 return inner.split(",") if inner else [] 

881 

882 def process(value): 

883 if value is None: 

884 return None 

885 return super_rp(handle_raw_string(value)) 

886 return process 

887 

888E.g.:: 

889 

890 Table( 

891 'mydata', metadata, 

892 Column('id', Integer, primary_key=True), 

893 Column('data', ArrayOfEnum(ENUM('a', 'b, 'c', name='myenum'))) 

894 

895 ) 

896 

897This type is not included as a built-in type as it would be incompatible 

898with a DBAPI that suddenly decides to support ARRAY of ENUM directly in 

899a new version.

1289class ENUM(sqltypes.NativeForEmulated, sqltypes.Enum): 

1290 

1291 """PostgreSQL ENUM type. 

1292 

1293 This is a subclass of :class:`.types.Enum` which includes 

1294 support for PG's ``CREATE TYPE`` and ``DROP TYPE``. 

1295 

1296 When the builtin type :class:`.types.Enum` is used and the 

1297 :paramref:`.Enum.native_enum` flag is left at its default of 

1298 True, the PostgreSQL backend will use a :class:`.postgresql.ENUM` 

1299 type as the implementation, so the special create/drop rules 

1300 will be used. 

1301 

1302 The create/drop behavior of ENUM is necessarily intricate, due to the 

1303 awkward relationship the ENUM type has in relationship to the 

1304 parent table, in that it may be "owned" by just a single table, or 

1305 may be shared among many tables. 

1306 

1307 When using :class:`.types.Enum` or :class:`.postgresql.ENUM` 

1308 in an "inline" fashion, the ``CREATE TYPE`` and ``DROP TYPE`` is emitted 

1309 corresponding to when the :meth:`.Table.create` and :meth:`.Table.drop` 

1310 methods are called:: 

1311 

1312 table = Table('sometable', metadata, 

1313 Column('some_enum', ENUM('a', 'b', 'c', name='myenum')) 

1314 ) 

1315 

1316 table.create(engine) # will emit CREATE ENUM and CREATE TABLE 

1317 table.drop(engine) # will emit DROP TABLE and DROP ENUM 

1318 

1319 To use a common enumerated type between multiple tables, the best 

1320 practice is to declare the :class:`.types.Enum` or 

1321 :class:`.postgresql.ENUM` independently, and associate it with the 

1322 :class:`.MetaData` object itself:: 

1323 

1324 my_enum = ENUM('a', 'b', 'c', name='myenum', metadata=metadata) 

1325 

1326 t1 = Table('sometable_one', metadata, 

1327 Column('some_enum', myenum) 

1328 ) 

1329 

1330 t2 = Table('sometable_two', metadata, 

1331 Column('some_enum', myenum) 

1332 ) 

1333 

1334 When this pattern is used, care must still be taken at the level 

1335 of individual table creates. Emitting CREATE TABLE without also 

1336 specifying ``checkfirst=True`` will still cause issues:: 

1337 

1338 t1.create(engine) # will fail: no such type 'myenum' 

1339 

1340 If we specify ``checkfirst=True``, the individual table-level create 

1341 operation will check for the ``ENUM`` and create if not exists::
